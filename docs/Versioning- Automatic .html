Automatic Data Protection

A feature that automatically maintains previous versions of a blob when it's overwritten or deleted.

  • Automatic Capture: You don't manually create versions; Azure does it automatically when specific actions occur.
  • Historical Record: Each version is a full, read-only snapshot of the blob at a specific point in time.
  • Recovery: Allows you to view, download, or restore previous states of your data.
  • Protection: Guards against accidental overwrites or deletions by preserving prior states.

Actions Creating New Versions

Specific operations on a blob cause its current state to become a previous version.

On Overwrite:

Blob (Current Data V1)
Overwrite Operation
(e.g., Put Blob, Put Block List, Copy Blob)

+
Blob (Current Data V2)

When a blob is overwritten, the state just before the overwrite becomes a new version.

On Delete:

Blob (Current Data V2)
Delete Operation
Version 2 (Old Data, Marked Deleted*)
+
No Current Blob

When the current blob is deleted, its state just before deletion becomes a new version. (*If Soft Delete is also enabled, this version is technically soft-deleted).

  • Overwrite Triggers: `Put Blob`, `Put Block List`, `Copy Blob` (when destination exists).
  • Delete Trigger: `Delete Blob`.
  • Deleting a specific version does NOT create a new version.

Core Characteristics

  • Automatic: No manual steps needed to create versions once enabled.
  • Complete History: Captures all overwrites and deletions of the current blob.
  • Immutable Versions: Once created, a version's content cannot be changed.
  • Unique Version IDs: Each version is identified by a timestamp. The current blob has no version ID.
  • Full Objects: Each version is a complete copy of the blob at that time, not just a delta.

Why Enable It?

  • Accidental Overwrite/Delete Recovery: Easily restore a previous state if the current blob is modified or deleted unintentionally.
  • Audit Trail / History: Provides a historical record of changes to a blob.
  • Foundation for Other Features: Required for Object Replication and works alongside Point-in-Time Restore (for block blobs).

Important Points

  • Enablement: Must be enabled at the storage account level. Cannot be disabled once enabled (though future versions might not be created if interactions change).
  • Cost: Each version is a full blob and incurs storage costs based on its size and tier. You pay for all versions + the current blob.
  • Lifecycle Management Interaction: Use Lifecycle rules to manage the cost of versions by tiering older versions to cheaper tiers (Cool, Cold, Archive) or deleting them after a certain time (`daysAfterVersionCreation`).
  • Soft Delete Interaction: If Soft Delete is also enabled, deleting the *current* blob creates a version which is then soft-deleted. Deleting a *specific version* makes that version soft-deleted.
  • Object Replication Dependency: Required on both source and destination for Object Replication to function. Replication copies *new versions* created on the source.
Plan for version management using Lifecycle policies to avoid excessive storage costs from accumulated versions.
