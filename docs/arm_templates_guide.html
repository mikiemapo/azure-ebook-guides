JSON Sections Overview

The main building blocks of an ARM template file.

"$schema": Validation schema URL (Intellisense).
"contentVersion": Your template version (Required).
"parameters": Inputs for customization during deployment.
"variables": Internal values/expressions for reuse.
"functions": Custom, reusable expressions (Less common now with Bicep).
"resources": **Core:** Defines the Azure resources to deploy. (Required).
"outputs": Values returned after successful deployment. 

{
  "$schema": "...",
  "contentVersion": "1.0.0.0",
  "parameters": { /* Deployment Inputs */ },
  "variables": { /* Internal Values */ },
  "functions": [ /* Custom Logic */ ],
  "resources": [ /* Azure Resources Defined Here */ ],
  "outputs": { /* Deployment Results */ }
}

Understanding the Sections

Focusing on Parameters, Variables, Resources, Outputs.

  • Parameters: Make templates flexible. Use for names, SKUs, locations, credentials (secureString/KeyVault refs). Defined with `type`, `metadata`, `defaultValue`, `allowedValues`.
  • Variables: Construct complex values once, reuse many times. Good for naming conventions (`uniqueString`, `concat`). Not for external input.
  • Functions: Use `[functionName(arguments)]`. Essential functions: `resourceGroup()`, `subscription()`, `concat()`, `resourceId()`, `reference()`, `listKeys()`, conditions (`if`), loops (`copy`).
  • Resources: Array defining each Azure resource. Key properties: `type`, `apiVersion`, `name`, `location`, `properties`, `sku`, `kind`, `tags`, `dependsOn`.
  • Outputs: Return deployment results like Public IPs, FQDNs, connection strings. Defined with `type`, `value`.

Separating Inputs

Manage deployment inputs cleanly using `.parameters.json` files.

  • Purpose: Provide values for parameters defined in the main ARM template.
  • Best Practice: Use separate parameter files for different environments (Dev, Test, Prod).
  • Structure: Simple JSON mapping parameter names to values: `{ "$schema": "...", "contentVersion": "...", "parameters": { "paramName": { "value": "paramValue" } } }`.
  • Secrets: Reference Key Vault secrets for sensitive values instead of storing them directly: `{ "paramName": { "reference": { "keyVault": { "id": "..." }, "secretName": "..." } } }`.

Incremental vs. Complete (AZ-104 Crucial!)

Understand how ARM handles resources during deployment.

Incremental (Default)

Adds resources defined in template.
Updates existing resources if changed.
Leaves resources in RG untouched if not in template.

Complete

Adds resources defined in template.
Updates existing resources if changed.
Deletes resources in RG that are NOT in the template.
Use with extreme caution!

Select the mode during deployment (e.g., `--mode Complete` in CLI/PS).

Targeting Deployments

Deploy templates at different levels in Azure.

  • Tenant: Global settings, Management Groups.
  • Management Group: Policies/RBAC across Subscriptions.
  • Subscription: Resource Groups, Policies/RBAC in a Sub.
  • Resource Group: Most common scope for deploying workload resources (VMs, DBs, etc.).

Specify the scope when using CLI/PowerShell deployment commands.

Creating Templates

Recommended tools for writing ARM templates.

  • VS Code + ARM Tools Extension: Best choice. Provides Intellisense, snippets, validation, schema awareness.
  • Azure Portal: Good for exporting existing resources ("Export template") or simple deployments. Not ideal for complex authoring.
  • Visual Studio: Azure Resource Group project provides integrated experience.
  • Text Editor: Possible but lacks helpful features and is error-prone.

Running Your Templates

How to initiate the deployment process.

  • Azure Portal: "Deploy a custom template". User-friendly for manual runs.
  • Azure CLI: `az deployment group/sub/mg/tenant create`. Scriptable.
  • Azure PowerShell: `New-AzResourceGroup/Subscription/ManagementGroup/TenantDeployment`. Scriptable.
  • CI/CD Pipelines: Automated deployments via Azure DevOps, GitHub Actions.
  • Azure SDKs: Integrate into custom applications (Python, .NET, etc.).
  • Azure Cloud Shell: Convenient browser-based CLI/PowerShell access.

A Simpler DSL

Abstracts ARM JSON for a cleaner authoring experience.

  • Concise Syntax: Easier to read and write than JSON.
  • Transpiles to ARM JSON: Compiles to standard ARM templates; uses ARM engine.
  • Improved Tooling: Better type safety, validation, modules.
  • Recommended by Microsoft for new IaC development on Azure.

Learn Bicep if you find ARM JSON verbose (`.bicep` files).

Storing Templates in Azure

Share and manage ARM templates as Azure resources.

  • Azure Resource: Store templates directly within Azure.
  • Versioning: Manage different versions of your templates.
  • RBAC Control: Secure access using standard Azure roles.
  • Simplified Sharing: Easier organizational sharing and standardization.
