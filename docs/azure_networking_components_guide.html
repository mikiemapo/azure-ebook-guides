L3/L4 Traffic Filtering

Filters network traffic to and from Azure resources in an Azure Virtual Network (VNet).

Internet / On-Prem
NSG Rules Allow Port 80 Inbound Deny Port 22 Inbound
VM NIC / Subnet
  • Rule-Based: Contains security rules that allow or deny inbound/outbound traffic based on Source/Destination IP, Port, and Protocol (TCP/UDP/ICMP/Any).
  • Scope: Can be associated with Network Interfaces (NICs) or Subnets. If associated with both, subnet rules apply first, then NIC rules.
  • Stateful (Mostly): For inbound rules that allow traffic, NSGs automatically allow the return traffic. Outbound rules are also stateful in this way. *However, for established TCP connections initiated outbound, return traffic is allowed regardless of inbound rules.*
  • Default Rules: Includes default rules (e.g., AllowVNetInBound, AllowAzureLoadBalancerInBound, DenyAllInBound). Custom rules have higher priority.

Use Cases: Basic firewalling for VMs and subnets, segmenting traffic within a VNet.

Implications: Essential for basic network security. Misconfiguration can block legitimate traffic or allow unwanted access.

Centralized Network Security

A managed, cloud-native, stateful firewall service offering network and application level protection.

Spoke VNets / On-Prem
Azure Firewall (Hub VNet)
Internet / PaaS
  • Stateful Firewall as a Service: Centrally manages and logs all traffic.
  • Policy Types:
    • Network Rules: L3/L4 filtering (Source/Dest IP, Port, Protocol).
    • Application Rules: Filter outbound HTTP/S traffic to specific Fully Qualified Domain Names (FQDNs).
    • NAT Rules (DNAT): Translate inbound traffic to internal resources.
  • Threat Intelligence: Can block traffic to/from known malicious IPs/domains (Standard & Premium).
  • Premium SKU Features: TLS inspection, IDPS (Intrusion Detection/Prevention), URL filtering, Web categories.
  • High Availability & Scalability: Built-in.

Use Cases: Centralized egress/ingress filtering for VNets (Hub-Spoke), protecting against threats, filtering outbound web traffic.

Implications: More advanced and centralized than NSGs. Higher cost. Requires proper routing (UDRs) to force traffic through it.

Secure Cross-Premises Connectivity

Establishes encrypted tunnels between your Azure VNet and on-premises networks (Site-to-Site) or individual clients (Point-to-Site) over the public internet.

On-Premises Network / User
Azure VPN Gateway (in GatewaySubnet)
Azure VNet
  • Site-to-Site (S2S): Connects an on-premises VPN device to an Azure VPN Gateway. Uses IPsec/IKE.
  • Point-to-Site (P2S): Connects individual client computers to an Azure VNet. Uses SSTP, OpenVPN, or IKEv2.
  • VNet-to-VNet: Connect Azure VNets (can be cross-region).
  • Encrypted Tunnels: Secures traffic over the public internet.
  • SKUs & Performance: Various SKUs offer different throughput and connection limits.

Use Cases: Hybrid cloud connectivity, secure remote access for users, connecting VNets. Often part of a DR strategy.

Implications: Essential for hybrid scenarios. Requires a `GatewaySubnet` in your VNet. Performance depends on internet links and gateway SKU.

Secure RDP/SSH via Portal

A fully managed PaaS service that provides secure and seamless RDP and SSH connectivity to your virtual machines directly from the Azure portal over TLS, without needing public IPs on VMs.

Admin User
Azure Portal (Browser)
Azure Bastion Host (AzureBastionSubnet)
VM (Private IP Only) No Public IP
  • No Public IP Exposure: VMs do not require public IP addresses for RDP/SSH access.
  • TLS Encrypted Session: All traffic between your browser and Bastion, and Bastion to VM, is over TLS/SSL.
  • Azure AD Integration: Can integrate with Azure AD for authentication to VMs (depending on VM OS/config).
  • Deployed in VNet: Requires a dedicated subnet named `AzureBastionSubnet`.
  • Managed Service: No need to manage bastion host VMs yourself.

Use Cases: Securely access VMs for management without exposing them to the internet. Reduce attack surface.

Implications: Improves security posture. Cost associated with the Bastion service. Simplifies RDP/SSH access management.

Private Access to PaaS/Azure Services

Enables you to access Azure PaaS services (like Azure Storage, SQL Database) and your own services using a private IP address from your VNet. Traffic stays on the Microsoft network.

VM in VNet
Private IP
Private Endpoint (in VNet Subnet)
Azure PaaS Service (e.g., Storage, SQL)
  • Private IP: Assigns a private IP from your VNet's address space to the PaaS service.
  • No Public Internet Exposure: Traffic to the PaaS service does not traverse the public internet.
  • Azure Private Link: Private Endpoints are a component of Azure Private Link service.
  • DNS Integration: Requires proper DNS configuration (e.g., Private DNS Zones) to resolve the PaaS service's FQDN to the private IP.

Use Cases: Securely accessing PaaS services from within a VNet or connected on-premises networks, data exfiltration protection.

Implications: Enhances security for PaaS services. Adds DNS management complexity. Cost for the Private Endpoint and data processed.

Layer 7 Load Balancer & Web Protection

A web traffic load balancer that enables you to manage traffic to your web applications. The Web Application Firewall (WAF) SKU provides protection against common web exploits.

User / Client
App Gateway
( WAF if enabled)
Web Server 1
Web Server 2
  • Layer 7 Load Balancing: Makes routing decisions based on HTTP/HTTPS attributes (URL path, host headers, cookies).
  • SSL/TLS Termination: Can offload SSL/TLS encryption/decryption from backend servers.
  • URL-based Routing: Route traffic to different backend pools based on URL paths.
  • Cookie-based Session Affinity: Directs client requests to the same backend server.
  • Web Application Firewall (WAF): (Optional SKU) Protects against common web vulnerabilities (e.g., SQL injection, XSS) based on OWASP Core Rule Sets.

Use Cases: Load balancing HTTP/S traffic for web applications, SSL offloading, protecting web apps with WAF.

Implications: Provides advanced web traffic management and security. More complex and costly than basic load balancers.

Custom Traffic Paths

User-Defined Routes (UDRs) allow you to override Azure's default routing behavior and control how network traffic is routed between subnets, VNets, or to/from on-premises via a Network Virtual Appliance (NVA).

Subnet A
Route Table (UDR)
(Route to NVA)
NVA / Azure Firewall
Internet
  • Override Default Routing: By default, Azure routes traffic based on its system routes. UDRs let you define custom next hops.
  • Next Hop Types: Virtual Appliance (NVA), Virtual Network Gateway, Internet, VNet Peering, None.
  • Association: Route tables are associated with subnets. All traffic leaving that subnet will be evaluated against the routes in the table.
  • Forcing Traffic: Commonly used to force traffic through an NVA (like Azure Firewall or a third-party firewall) for inspection.

Use Cases: Implementing Hub-Spoke topologies with centralized NVAs, forcing internet-bound traffic through a security appliance, custom routing between VNets.

Implications: Powerful for custom network architectures. Misconfiguration can lead to traffic black-holing or routing loops. Essential for NVA deployments.
