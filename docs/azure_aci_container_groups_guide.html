Lightweight, Executable Package

A container is a standard unit of software that packages up code and all its dependencies so the application runs quickly and reliably from one computing environment to another.

Software ContainerCode + Runtime + System Tools + Libraries + Settings
Shipping ContainerSelf-contained, Portable
  • Includes Everything Needed: Code, runtime (e.g., .NET, Node.js), system tools, system libraries, and settings.
  • Lightweight: Unlike VMs, containers virtualize the OS (share the host OS kernel) instead of virtualizing hardware.
  • Standalone & Executable: Can run independently.
  • Consistent Operation: Runs the same regardless of where it's deployed (dev laptop, on-prem, cloud).

Analogy Implication: Just like a physical shipping container standardizes transport, software containers standardize application packaging and deployment.

Advantages for Cloud Workloads

Azure provides robust support for containers, leveraging their inherent benefits for cloud-native applications.

  • Faster to Deploy: Containers start much quicker than VMs.

Azure Services: Azure offers Azure Container Instances (ACI), Azure Kubernetes Service (AKS), Web App for Containers, and Azure Container Registry (ACR) to support containerized workloads.

Simplest Way to Run Containers in Azure (PaaS)

ACI allows you to run Docker containers directly in Azure without needing to provision or manage any underlying virtual machines or complex orchestrators.

  • Serverless Containers: No VM infrastructure to manage. Azure handles the underlying hosts.
  • Fast Startup: Containers can start in seconds.
  • Simple Use Cases: Ideal for simple applications, task automation, build jobs, data processing, or bursting from AKS.

Portal Workflow for ACI Creation

Based on your transcript, a typical flow for deploying an ACI via the Azure portal.

  1. Navigate to "Container Instances" in Azure portal & click "Create".
  2. Basics Tab:
    • Select/Create Resource Group.
    • Enter unique Container Name.
    • Choose Region.
    • (Optional) Availability Zones.
  3. Container Tab (Image Settings):
    • Image Source: Quickstart, Azure Container Registry (ACR), Docker Hub/Other Public.
    • OS Type: Linux or Windows.
    • Image: Specify the image (e.g., `mcr.microsoft.com/azuredocs/aci-helloworld`).
  4. Container Tab (Size Settings - per container if multiple):
    • CPU Cores (e.g., 1-4).
    • Memory (e.g., 1.5GB - 14GB).
    • GPU (Optional).
  5. Networking Tab:
    • Networking Type: Public (internet accessible) or Private (VNet only).
    • DNS Name Label: (For Public) Creates a FQDN like `yourlabel.region.azurecontainer.io`.
    • Ports: Define which container ports to expose (e.g., TCP 80).
  6. Review settings and click "Create". Deployment is usually very fast (~30 seconds - few minutes).

AZ-104 Focus: Knowing the key configuration settings: RG, Name, Region, Image Source, OS Type, CPU/Memory, Networking (Public/Private IP, DNS label, Ports).

The ACI Deployment Unit

The top-level resource in Azure Container Instances. A container group is a collection of containers that get scheduled on the same host machine.

Container Group (Single Host, IP, Lifecycle)
App
Sidecar
  • Same Host Machine: All containers within a single group run on the same underlying Azure-managed VM node.
  • Shared Lifecycle: Containers in a group start, stop, and are deleted together.
  • Shared Resources: They share a network namespace (same IP address, port space), and can optionally share storage volumes (e.g., Azure Files). CPU and Memory are allocated to the group and shared.
  • "Pod-like" (Simplified): Conceptually similar to a Kubernetes Pod but much simpler and specific to ACI.

Implication: Allows co-locating tightly coupled containers that need to share resources or communicate via localhost.

Defining Characteristics

  • Single IP Address & DNS Name: All containers in the group share one public IP (if configured) and one DNS name label.
  • Exposed Ports: You can expose different ports from different containers within the group on that single IP.
  • Shared Volumes: Can mount Azure File Shares as volumes, accessible by all containers in the group.
  • Internal Communication: Containers within the same group can communicate with each other using `localhost:`.
  • Restart Policy: Define what happens if a container stops (`Always`, `OnFailure`, `Never`).

Multi-Container Scenario

Scenario: A web application with a frontend container and a logging sidecar container.

Container Group (Public IP:80)
Web FrontendPort 80 (Exposed)
Logging SidecarPort 5000 (Internal)
Web Frontend uses localhost:5000 to send logs to Sidecar. Both can access a shared Azure File Share.
  • **Container A (Frontend):** Serves web content on port 80 (exposed publicly).
  • **Container B (Logging Sidecar):** Collects logs from the frontend, perhaps listening on an internal port like 5000.
  • Frontend sends logs to `localhost:5000` to reach the sidecar.
  • Both might mount a shared Azure File Share for persistent data or shared configuration.

How to Create Container Groups

  • Azure Portal (GUI): Good for quick tests, demos, and visual configuration.
  • ARM Templates (JSON): For Infrastructure-as-Code (IaC), repeatable deployments, enterprise scenarios.
  • YAML Files (with CLI/PS): Declarative definitions, similar to Kubernetes style, often used with `az container create --file`.
  • Azure CLI: `az container create ...` (scripting, automation).
  • Azure PowerShell: `New-AzContainerGroup ...` (scripting, automation).
  • Bicep: Newer DSL that transpiles to ARM JSON, offering simpler syntax for IaC.

Key Concepts for the Exam

Containers vs VMs ACI Container Groups Deployment Top
  • ACI vs. AKS: Know that ACI is for simpler, per-container/group deployments without orchestration, while AKS is for full Kubernetes orchestration.
  • Container Groups: Understand they are the deployment unit, share resources (IP, network, lifecycle, optional volumes).
  • Port Exposure: How ports are exposed publicly vs. used for internal communication within a group.
  • Volume Mounting: Primarily Azure File Shares for persistent data in ACI.
  • Image Sources: Quickstart, ACR, Docker Hub/Public.
  • Networking: Public vs. Private IP, DNS name label.
  • Cost Model: Per-second billing for CPU/memory.
  • Scaling: ACI itself is not for horizontal auto-scaling of a single application like AKS; you scale by deploying more/larger container groups or use it for burst from AKS.

ACI Scaling Weak Areas Memorize

  • No in-place scaling: You cannot adjust instance count of a running group.
  • Redeploy to scale: Scaling means creating a new container group with desired resources.
  • No built-in load balancing or auto-scaling: Use AKS / orchestrators for dynamic scale + traffic distribution.
