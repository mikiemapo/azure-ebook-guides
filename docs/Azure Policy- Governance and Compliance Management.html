Azure Policy: Governance & Compliance (AZ-104)

Master Azure Policy for cloud resource governance and compliance. Focused on AZ-104 essentials. Click any section header to expand and explore core concepts, components, and best practices.

Core Idea

  • A service for enforcing standards and ensuring compliance across Azure resources.

Key Uses

  • Evaluate resource compliance.
  • Enforce organizational rules (e.g., resource tagging, allowed regions).
  • Automatically remediate non-compliant resources (for certain effects).
AZ-104 Focus: Recognize Azure Policy as a governance and compliance enforcement tool for consistent cloud environments.

Three Main Objects

  • Policy Definition: A JSON rule that defines what to evaluate and the effect if conditions are met.
  • Initiative Definition (policySet): A group of policy definitions collected for common governance goals (e.g., "ISO 27001 Compliance").
  • Assignment: The act of applying a policy definition or initiative to a specific scope.
AZ-104 Tip: Be able to distinguish each component and its purpose. Policies are like individual rules, initiatives are rule sets, and assignments are where you apply them.

Scope

  • Defines the range of resources to which a policy assignment applies.
  • Hierarchy: Management Group → Subscription → Resource Group → Individual Resource. Policies assigned at higher levels apply to all child resources unless excluded.

Parameters

  • Variables defined within a policy definition or initiative.
  • Allow for reuse and flexibility by letting you inject different values (e.g., allowed regions, tag names) at assignment time.
AZ-104 Focus: Know how scope inheritance and parameters make policies reusable.

Common Effects

  • Deny: Blocks the creation or update of non-compliant resources.
  • Audit: Allows non-compliant resources but creates a warning event in the activity log and marks the resource as non-compliant in the compliance dashboard.
  • Modify: Automatically corrects non-compliance by altering resource properties during creation/update or for existing resources.
  • DeployIfNotExists (DIFE): Automatically deploys a template/resource if a related condition is not met (e.g., deploy a diagnostic setting if one isn't configured).
  • AuditIfNotExists (AIFE): Audits if a related resource or configuration isn't found.
  • Disabled: Deactivates the policy; it won't be evaluated.
Mnemonic: D-A-A-M-D → Deny, Audit, Append, Modify, DeployIfNotExists, Disabled.
AZ-104 Tip: Always start with the Audit effect during testing to understand the impact of your policy before enforcing it with Deny or Modify.

Remediation

  • The process of bringing existing non-compliant resources into compliance with policy requirements.
  • Can be automated for Modify, DeployIfNotExists, and AuditIfNotExists effects.
  • Manual remediation tasks can also be created.

Evaluation Triggers

  • Every 24 hours: Standard, automatic compliance evaluation cycle.
  • On resource changes: When a resource is created, updated, or deleted.
  • On policy/assignment changes: When a policy definition, initiative, or assignment is created or modified.
AZ-104 Tip: Know that remediation actions are triggered for existing non-compliant resources, not just new ones. Understand the frequency and events that trigger policy evaluations.

RBAC (Role-Based Access Control)

  • Controls who can do what within Azure (manages user actions and permissions).

Azure Policy

  • Controls what resources must look like and their configurations (ensures resource compliance).

Example

  • RBAC says: "You (as a user) can deploy Virtual Machines."
  • Policy says: "Virtual Machines must be deployed in the 'East US' region only." (or tagged appropriately).
AZ-104 Tip: Expect scenario-based questions requiring you to differentiate between RBAC (user permissions) and Policy (resource configuration/state enforcement). They are complementary governance layers.

Tips for Real-World Use

  • Start with Audit mode: Deploy policies in audit mode first to understand their impact without blocking operations.
  • Use Initiatives: Group related policies into initiatives even if it's just one policy initially, for future scalability and easier management.
  • Manage policies as code: Use ARM templates, Bicep, Azure CLI, or PowerShell for version control and CI/CD.
  • Align to organizational hierarchy: Apply policies at the highest possible scope (e.g., Management Group) for consistency.
  • Use built-in policies as a starting point.
Exam Tip: Policy-as-code and phased rollout (Audit before Deny/Modify) are key concepts emphasized for exam scenarios.

Integrates With

  • Azure Arc: Extends Azure Policy's governance capabilities to resources outside Azure (on-premises servers, other cloud providers).
  • AKS (Azure Kubernetes Service), Key Vault, VNet Manager: Provides deeper integration for specific resource types, allowing policy to target and manage their specific configurations.
  • Microsoft Defender for Cloud: Integrates for security compliance benchmarks and recommendations.
  • Policy Insights: A dedicated blade within Azure Policy for monitoring and analyzing compliance results.
Note: Policy extends beyond Azure using Arc for hybrid and multi-cloud governance.
AZ-104 Tip: Understand how Azure Policy extends its reach to hybrid/multi-cloud environments via Azure Arc. Be aware of its integration with key Azure services.

Key Takeaways for the Exam

  • Policy's Purpose: Governance and Compliance. Enforce organizational standards, assess resource compliance, automate remediation.
  • Core Objects: Policy Definition (the rule), Initiative Definition (group of rules), Assignment (applies policy/initiative to a scope).
  • Scope & Parameters: Policies apply hierarchically (Mgmt Group > Subscription > RG > Resource). Parameters allow flexibility at assignment time.
  • Effects: Understand Deny, Audit, Modify, DeployIfNotExists, AuditIfNotExists. Start with Audit for testing.
  • Evaluation: Every 24 hours, plus on resource/policy changes. Remediation brings existing resources into compliance.
  • Policy vs. RBAC: Policy manages resource state. RBAC manages user actions. They are complementary.
  • Best Practices: Phased rollout (Audit first), manage as code, use initiatives, align with hierarchy.
  • Integrations: Extends to hybrid/multi-cloud via Azure Arc. Integrates with AKS, Key Vault, Defender for Cloud, etc.
Exam Mnemonic: P-I-A-R → Policy → Initiative → Assignment → Remediation.
Final Exam Tip: Look for keywords in scenarios! "Enforce standards," "compliance," "non-compliant resources" point to Azure Policy. "Who can access," "permissions" point to RBAC.
