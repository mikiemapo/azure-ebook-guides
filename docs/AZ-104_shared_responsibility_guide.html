Understanding the Shared Responsibility Framework

  • What is it? The Azure Shared Responsibility Model defines which security and operational tasks are handled by Microsoft and which are the customer's responsibility.
  • Core Principle: As you move from IaaS to PaaS to SaaS, Microsoft takes on more responsibility, but the customer always retains certain responsibilities.
  • Why it matters: Understanding where responsibility lies is critical for security, compliance, and incident response in Azure environments.
Analogy: Think of cloud service models like housing:
  • IaaS (Landlord): You rent the building and land, but you manage everything inside—furniture, maintenance, security.
  • PaaS (Apartment): The landlord handles building maintenance and infrastructure; you manage your apartment contents.
  • SaaS (Hotel): Everything is managed for you; you just bring your personal items and data.

Responsibility Spectrum

IaaS → PaaS → SaaS

⬇️

More Control = More Responsibility

As you move right on the spectrum, Microsoft assumes more responsibility, and customers have less to manage.

AZ-104 Focus: Recognize how control and responsibility shift depending on the service model (IaaS, PaaS, SaaS). Know what you manage vs. what Azure manages in each model.

Data, Identity, Endpoints, Access

No matter the service model, the customer always retains responsibility for these critical areas:

  • Data Protection: Securing, classifying, and managing your data. This includes encryption at rest and in transit where applicable.
  • Identity & Access Management (IAM): Managing user identities, role assignments, multi-factor authentication (MFA), and access policies.
  • Endpoint Security: Protecting devices that access Azure resources (laptops, mobile devices, etc.).
  • Accounts & Access: Managing user accounts, permissions, and ensuring principle of least privilege.
Mnemonic: D.I.A.D.
Data | Identity | Access | Devices
These four areas are always your responsibility, no matter which Azure service model you use.
Example: Even when using Azure SQL Database (PaaS), you must:
  • Configure firewall rules and Network Security Groups (NSGs)
  • Enable Multi-Factor Authentication (MFA) for database admins
  • Manage data encryption keys and access policies
  • Control who has read/write access to the database
AZ-104 Focus: These responsibilities never shift, even in SaaS. Be prepared to identify customer-owned security controls in any service model scenario.

Platform, Physical, and Core Infrastructure

Microsoft handles the foundational layers of Azure's infrastructure, including:

  • Physical Datacenters: Physical security, power, cooling, and hardware maintenance.
  • Core Network: Azure's global network backbone, routers, and connectivity infrastructure.
  • Host Infrastructure: Physical servers, storage, and compute hardware.
  • Hypervisor: The virtualization layer that enables multi-tenancy and VM isolation.
  • Platform Services: For PaaS and SaaS, Microsoft manages the underlying runtime, middleware, and operating system updates.

Azure Layers under Microsoft Management

Physical Layer: Datacenters, racks, power, cooling

Network Layer: Azure backbone, edge routers, CDN

Compute Layer: Servers, storage arrays, hypervisor

Platform Layer (PaaS/SaaS): OS patching, runtime, middleware

Compliance Standards: Microsoft maintains numerous certifications and compliance standards:
  • SOC 1, 2, 3: Service organization controls
  • ISO 27001: Information security management
  • FedRAMP: Federal risk and authorization management
  • HIPAA/HITECH: Healthcare data protection
  • GDPR: European data privacy regulation

Microsoft is responsible for maintaining these certifications for the platform infrastructure.

AZ-104 Focus: Understand the scope of Microsoft's platform security and compliance responsibilities. Know that physical security and core infrastructure are always Microsoft's responsibility.

IaaS vs PaaS vs SaaS Responsibility Split

The shared responsibility model varies significantly based on which cloud service model you use:

Component IaaS PaaS SaaS
Data & Classification Customer Customer Customer
Identity & Access Customer Customer Customer
Applications Customer Customer Microsoft
Operating System Customer Microsoft Microsoft
Network Controls Shared Shared Microsoft
Runtime/Middleware Customer Microsoft Microsoft
Hypervisor Microsoft Microsoft Microsoft
Physical Infrastructure Microsoft Microsoft Microsoft

IaaS (Infrastructure as a Service)

  • Customer Manages: Virtual machines, operating systems, applications, data, network security groups (NSGs), firewall rules, patching, and updates.
  • Microsoft Manages: Physical servers, storage, networking infrastructure, hypervisor, datacenter security.
  • Example: Azure Virtual Machines, Azure Virtual Network

PaaS (Platform as a Service)

  • Customer Manages: Applications, data, application-level security, access controls.
  • Microsoft Manages: Operating system, runtime environment, middleware, patching, infrastructure, network security.
  • Example: Azure App Service, Azure SQL Database, Azure Functions

SaaS (Software as a Service)

  • Customer Manages: Data usage, access controls, user permissions, data classification.
  • Microsoft Manages: Everything else—application, runtime, OS, infrastructure, physical security.
  • Example: Microsoft 365, Dynamics 365, Azure AD (Entra ID)
AZ-104 Focus: Be able to identify management boundaries for each service model. Know which components shift from customer to Microsoft as you move from IaaS → PaaS → SaaS.

Summary Bullets for Exam Prep

  • Customer Always Manages: Data, identity, user access, endpoints—no matter the service model.
  • Microsoft Always Manages: Physical infrastructure, datacenters, hypervisor, core networking.
  • IaaS: Customer manages OS, applications, and most security configurations. Microsoft handles infrastructure.
  • PaaS: Microsoft manages OS and runtime; customer handles apps and data.
  • SaaS: Microsoft manages everything except data usage, access control, and user permissions.
  • Shared Responsibilities: Network controls and security are often shared between customer and Microsoft, especially in IaaS/PaaS.
  • Compliance: Microsoft maintains platform certifications (SOC, ISO, FedRAMP), but customers must ensure their data usage and configurations comply with regulations.
Exam Tip: When answering responsibility questions:
  1. Identify the service model (IaaS, PaaS, SaaS)
  2. Check if it's D.I.A.D. (Data, Identity, Access, Devices) → If yes, it's always customer
  3. Check if it's physical/infrastructure → If yes, it's always Microsoft
  4. For everything else, use the service model matrix to determine responsibility
