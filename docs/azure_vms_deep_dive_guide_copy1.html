What are Azure VMs? (IaaS)

Azure Virtual Machines (VMs) are on-demand, scalable computing resources that give you the flexibility of virtualization without having to buy and maintain the physical hardware that runs it. This is Infrastructure as a Service (IaaS).

YOU Manage Data, Apps, OS, Network Config
AZURE Manages Physical Hardware, Virtualization
  • Full Control: You choose the OS (Windows, various Linux distros), install software, configure networking, and manage storage.
  • Scalability: Easily change VM sizes (CPU, RAM, etc.) or deploy multiple VMs.
  • Global Reach: Deploy VMs in any Azure region worldwide.
  • Use Cases: Migrating on-premises servers ("lift-and-shift"), hosting applications requiring specific OS configurations, development/test environments, running databases, backend for web apps.

Implication: Provides maximum flexibility but comes with the responsibility of managing the guest OS, patching, software, and some networking aspects (as per Shared Responsibility Model).

Planning: Region & Availability

Choosing the right region and availability options is critical for application uptime.

  • Region: Geographic area where your VM is deployed. Choose based on user proximity, data sovereignty, service availability, and cost.
  • Single VM SLA: To get an SLA for a single VM, it must use Premium SSD or Ultra Disk for all its OS and data disks. (Typically 99.9% or higher based on disk type).
  • Availability Sets:
    • Protects against hardware failures within a single datacenter.
    • Spreads VMs across Fault Domains (FDs) - racks with independent power/network.
    • Spreads VMs across Update Domains (UDs) - groups that can be rebooted for planned maintenance together.
    • Typically provides 99.95% SLA if 2+ VMs are in an Availability Set.
  • Availability Zones:
    • Protects against entire datacenter failures within a region.
    • Physically separate locations within an Azure region, each with independent power, cooling, and networking.
    • You deploy VMs (or VMSS instances) across multiple zones.
    • Typically provides 99.99% SLA if VMs are deployed across zones.

VM Sizes & Series (Choosing Compute)

Azure offers a wide variety of VM sizes optimized for different workloads. Understanding the series helps you select the appropriate compute resources for performance and cost.

  • General Purpose (e.g., Dsv5, Dasv5, B-series, Av2-series):
    • Balanced CPU-to-memory ratio.
    • Ideal for testing, development, small to medium databases, and web servers.
    • B-series: Burstable VMs, cost-effective for workloads with typically low CPU usage but occasional peaks. Accrue credits when below baseline.
  • Compute Optimized (e.g., Fsv2-series):
    • High CPU-to-memory ratio.
    • Best for medium-traffic web servers, network appliances, batch processes, and application servers.
  • Memory Optimized (e.g., Esv5, Easv5, M-series):
    • High memory-to-CPU ratio.
    • Great for relational database servers (SQL, MySQL), medium to large caches, and in-memory analytics.
    • M-series: Offer very high memory (up to 12TB), for largest in-memory workloads like SAP HANA.
  • Storage Optimized (e.g., Lsv3, Lasv3-series):
    • High disk throughput and IOPS. Features fast, large local NVMe storage.
    • Ideal for Big Data, SQL and NoSQL databases (Cassandra, MongoDB), data warehousing, and large transactional databases.
  • GPU Optimized (N-series):
    • Include NVIDIA GPUs (e.g., T4, A10, A100).
    • Designed for compute-intensive, graphics-intensive, and AI/ML workloads, video editing, rendering, deep learning.
  • High Performance Compute (HPC) (e.g., HB-series, HC-series):
    • Optimized for highest-end computational workloads, often with low-latency RDMA networking (InfiniBand).
    • Used for financial modeling, weather simulation, engineering simulations.
  • Resizing VMs: You can change the size of an existing VM using az vm resize (CLI) or Update-AzVM with Set-AzVMHardwareProfile (PowerShell), or via the portal. The VM typically restarts. Ensure the new size is available in the region/Availability Zone.

AZ-104 Focus: Understanding IaaS fundamentals; difference between Availability Sets and Availability Zones with SLA implications (FDs/UDs for sets); knowing VM series purposes (D for general, E for memory, F for compute, N for GPU, L for storage IO); sizing based on workload; B-series (burstable) concept; resizing VMs.

Implication: Proper planning (region, availability option, VM size/series) is foundational for balancing performance, cost, and resilience. Over-provisioning wastes money; under-provisioning hurts performance.

VM Storage: OS Disks, Data Disks, Types & More

Understanding the storage options available for your Azure VMs.

  • OS Disk: Every VM has one OS disk (Windows or Linux). Managed Disks are recommended. Default size varies (e.g., ~127GB for Windows). Max 2TB (MBR) or 4TB (GPT, Gen2 VMs usually).
  • Data Disks: Optional additional disks for data storage. Can be attached/detached. Must be initialized and formatted within the OS. Max 32TB per disk (P50/E50 or larger).
  • Managed Disk Types (Performance & Cost):
    • Standard HDD: Lowest cost, for dev/test, backups, less critical workloads. Higher latency.
    • Standard SSD: Consistent performance for web servers, light app servers, dev/test. Better latency than HDD.
    • Premium SSD: High-performance, low-latency for I/O-intensive production (SQL Server, Dynamics). Higher IOPS/throughput.
    • Ultra Disk: Highest performance Azure disk, sub-millisecond latency for top-tier databases (SAP HANA, etc.). Dynamically configurable performance.
  • Temporary Disk: Present on the physical host (not durable, SSD-based). For temporary data like page files, swap files, SQL Server TempDB. Data lost on deallocation, resize, host failure, or migration. Size varies by VM size. Do NOT store persistent data here.
  • Disk Caching (Host Caching): Can improve performance.
    • OS Disk: Default is Read/Write.
    • Data Disks: Options are None, ReadOnly, ReadWrite. ReadOnly recommended for read-heavy workloads (like SQL data files). ReadWrite for balanced workloads (but risk of data loss on host failure if app doesn't handle write-back caching).

Securing VM Disks: Azure Disk Encryption (ADE)

This section covers a lab flow: 1. Deploying/Preparing a Windows VM. 2. Enabling Azure Disk Encryption (ADE) using PowerShell & Key Vault. 3. Subsequently using an ARM Quickstart template from the Azure portal to disable/decrypt encryption.

Lab Workflow Summary:

  1. Windows VM (`fmdata-vm01`) created/prepared (e.g., via `New-AzVm`).
  2. Initial Disk Encryption Status Checked: `Get-AzVmDiskEncryptionStatus` (Showed "NotEncrypted").

  7. Final Status Check via PowerShell: `Get-AzVmDiskEncryptionStatus` (Showed "OsVolumeEncrypted : NotEncrypted" with "Disable Encryption completed successfully").

VM Operating System Images

The operating system image is the starting point for your Azure VM, defining the OS and initial software configuration.

1. Azure Marketplace Images

  • Pre-configured OS images (Windows Server, various Linux distributions like Ubuntu, CentOS, RHEL) and application images (e.g., SQL Server on Windows, WordPress).
  • Provided by Microsoft, open-source communities, and third-party vendors.
  • Discoverable and deployable via Azure portal, CLI (az vm image list), PowerShell (Get-AzVMImageSku), and ARM templates.
  • Often include necessary drivers and optimizations for Azure.
  • May include software licensing costs (e.g., Windows Server, paid Linux distros) or be "Bring Your Own License" (BYOL) / "Bring Your Own Subscription" (BYOS) for certain vendor images.

2. Custom Images

  • Purpose: Create standardized VM deployments with your pre-installed applications, configurations, and security hardening. Reduces deployment time and ensures consistency.
  • Generalized vs. Specialized Images:
    • Generalized: Machine-specific information (computer name, user accounts) is removed using tools like Sysprep (Windows) or `waagent -deprovision` (Linux). Used as a template to create multiple new VMs. This is the common approach for custom image templates.
    • Specialized: A snapshot of a VM's OS disk without generalization. Creates an exact copy of the source VM, including its state and data. Useful for backups or migrating a specific server.
  • Creation Methods:
    • From an existing Azure VM (after generalizing).
    • Using Azure Image Builder (a managed service for building and distributing images).
    • Uploading a generalized VHD from on-premises.
  • Hands-on Lab: For practical steps on creating a custom image from a Windows VM, see the Lab Review: Windows VM, Web Server & Image Capture guide.

3. Azure Compute Gallery (formerly Shared Image Gallery)

  • A service to manage and share images at scale.
  • Allows grouping images, versioning, and replicating them to different Azure regions for faster local deployments.
  • Control access to images using Azure RBAC.
  • Share images across subscriptions and even different Azure AD tenants.
  • Supports both generalized and specialized images.

Key Considerations for Images

  • Licensing: Ensure compliance for OS and any included software. Marketplace images handle this transparently for base OS; for custom images, you are responsible.
  • VM Generation:
    • Gen1 VMs use BIOS-based boot.
    • Gen2 VMs use UEFI-based boot, support features like Trusted Launch, larger OS disks, and specific OS versions.
    • Ensure your chosen image is compatible with the VM generation you plan to use. Most modern Marketplace images support Gen2.
  • Image Updates: Custom images need to be periodically updated (patched) and new versions created to ensure security and include latest software.

VM Deployment Methods

Azure offers multiple ways to deploy VMs. Each method suits different needs, from quick tests to complex, automated infrastructure rollouts.

1. Azure Portal

  • Provides a graphical user interface (GUI).
  • Good for learning and one-off deployments.
  • Visually guides through configuration options.
  • Less ideal for repeatable or automated large-scale deployments.

2. Azure CLI (Command-Line Interface)

  • Cross-platform command-line tool.
  • Uses commands like az vm create.
  • Excellent for scripting and automation tasks.
  • Can create VMs with complex settings.
  • Initial learning curve can be steeper than the portal.

Example: az vm create --resource-group MyRG --name MyVM --image UbuntuLTS ...

3. Azure PowerShell

  • A command-line shell and scripting language.
  • Built on .NET; strong in Windows environments.
  • Uses cmdlets such as New-AzVm.
  • Powerful for automation and managing complex tasks.
  • Often preferred by administrators familiar with Windows.

Example: New-AzVm -ResourceGroupName MyRG -Name MyVM ...

4. ARM Templates (Azure Resource Manager)

  • Declarative JSON files. Define the desired state.
  • Define all resources (VMs, storage, network) & configs.
  • Ensures consistent and repeatable deployments.
  • Idempotent: Re-running results in the same state.
  • Foundation for Infrastructure as Code (IaC) in Azure.
  • Can be complex to author initially. Portal can export templates.

Deploy ARM templates using Portal, CLI (az deployment group create), or PowerShell.

5. Other Tools (e.g., Terraform, Bicep)

  • Third-party IaC tools like Terraform.
  • Azure Bicep: A simpler DSL for Azure deployments. Transpiles to ARM JSON.
  • Often used for multi-cloud or advanced IaC scenarios.

AZ-104 Focus: Understanding difference between Marketplace and custom images, the purpose of generalization, benefits of Azure Compute Gallery, VM generation compatibility. Familiarity with Portal, CLI, PowerShell for VM deployment; understanding ARM templates for repeatable IaC deployments.

Implication: Proper image selection (Marketplace vs custom, generalized vs specialized) ensures consistent and secure deployments. Choosing the right deployment method (Portal for learning, CLI/PowerShell for automation, ARM templates for IaC) balances efficiency with repeatability.

Connecting Your Virtual Machines

Understanding how Azure VMs connect to networks and the internet is crucial for accessibility, security, and communication with other resources.

1. Network Interface Cards (NICs)

  • A VM must have at least one NIC to connect to an Azure Virtual Network (VNet).
  • NICs are attached to a subnet within a VNet.
  • Primary NIC: The first NIC attached to a VM. Cannot be deleted unless the VM is deleted. Default gateway is assigned to this NIC.
  • Secondary NICs: Depending on VM size, you can attach multiple NICs.
    • Use cases: Network appliances, routing traffic between different subnets/VNets, separate management traffic.
  • NICs can be associated with Public IPs, Private IPs, and NSGs.

2. IP Addressing

  • Private IP Addresses:
    • Assigned from the address range of the subnet the NIC is in.
    • Used for communication within the VNet and connected networks (peered VNets, on-prem via VPN/ExpressRoute).
    • Can be Dynamic (default, assigned when VM starts) or Static (persists across VM restarts/deallocations). Static is recommended for servers providing services (e.g., domain controllers, DNS servers).
  • Public IP Addresses (PIP):
    • Allows inbound internet traffic to reach the VM (if NSG rules permit).
    • Enables outbound internet connectivity for the VM (by default).
    • Can be associated with a VM NIC, Load Balancer, Application Gateway, etc.
    • SKUs: Basic (older, fewer features, not zone-redundant) and Standard (recommended, zone-redundant option, more features, secure by default - requires NSG for inbound).
    • Assignment: Can be Dynamic (assigned at creation, may change if VM is stopped/deallocated - Basic SKU only) or Static (persists - Standard SKU is always static, Basic can be static). Static recommended for internet-facing services.

3. Domain Name System (DNS)

  • Azure-provided DNS (Default):
    • Azure automatically provides internal DNS name resolution for VMs within the same VNet and for public DNS resolution.
  • Custom DNS Servers:
    • Can configure custom DNS servers at the VNet level (applies to all VMs) or per NIC.
    • Needed for Active Directory domain resolution or specific on-premises DNS requirements.
  • Public DNS Name Label:
    • For Public IP addresses, you can configure a DNS name label (e.g., myvm.eastus.cloudapp.azure.com).
    • Creates an A record in Azure-managed DNS zones. Provides a friendly name to access the VM.

4. Network Security Groups (NSGs) - Recap

  • Act as a stateful firewall for VMs.
  • Contain inbound and outbound security rules to allow or deny traffic based on source/destination IP, port, and protocol.
  • Can be associated with a NIC or a subnet (or both). If both, rules are processed in order.
  • For a detailed understanding, refer to the Network Security Tools Guide.

AZ-104 Focus:

  • Assigning static vs. dynamic private IPs.
  • Purpose and SKUs of Public IPs; assigning DNS name labels.
  • When to use Azure-provided vs. custom DNS.
  • Role of NICs and how NSGs are associated for traffic filtering.

Day-to-Day VM Administration

Effectively managing your Azure VMs involves various operational tasks, from connecting and controlling state to monitoring and maintenance.

1. Connecting to VMs

  • Windows VMs (RDP):
    • Use Remote Desktop Protocol (RDP) clients. Requires RDP port (default 3389) to be open in NSG and OS firewall.
  • Linux VMs (SSH):
    • Use Secure Shell (SSH) clients (e.g., PuTTY, OpenSSH). Requires SSH port (default 22) to be open in NSG and OS firewall. Uses key-based or password authentication.
  • Azure Bastion (PaaS Service):
    • Provides secure RDP/SSH connectivity to your VMs directly in the Azure portal over SSL.
    • Eliminates the need to expose RDP/SSH ports publicly on your VMs. Deployed in your VNet.
    • Highly recommended for improved security.
  • Serial Console:
    • Text-based console access for troubleshooting boot issues or interacting with the OS when network connectivity is lost.

2. VM State Control

  • Start: Powers on the VM. Compute charges apply.
  • Stop (Deallocate): Shuts down the VM and releases compute resources (CPU, memory, dynamic IP). You stop paying for compute. Storage charges for disks continue.
  • Stop (Without Deallocation - OS Shutdown): Shuts down the OS but resources remain allocated. Compute charges continue. Rarely used unless specific scenarios require it.
  • Restart: Reboots the VM.
  • Delete: Removes the VM. Associated resources like NICs and OS disks are often deleted by default (configurable), but data disks are typically not.

3. Resizing VMs

  • Change the VM size (CPU, RAM, temp storage, max disks/IOPS) to meet performance or cost needs.
  • VM needs to be restarted (usually). Some sizes can be resized without restart if the current hardware cluster supports the new size.
  • If resizing to a size not available on the current hardware cluster, the VM might need to be stopped (deallocated) first, then resized.
  • Ensure the target VM size is available in the VM's region and Availability Zone (if applicable).

4. Monitoring VMs (Overview)

  • Metrics: Host-level metrics (CPU, Network, Disk) are collected by default.
  • Guest-Level Diagnostics: Enable via Azure Monitor Agent (AMA) or legacy Log Analytics Agent to collect OS performance counters and event logs.
  • Boot Diagnostics: For troubleshooting boot failures (screenshot, serial log).
  • Alerts: Configure alerts based on metrics or logs.
  • For comprehensive details, refer to the Azure Monitor & Maintenance Guide.

5. Patching & Updates

  • You are responsible for patching the guest OS and installed software on IaaS VMs.
  • Azure Update Management:
    • A service in Azure Automation to assess and deploy OS updates for Windows and Linux VMs across Azure and hybrid environments.
    • Requires Automation Account & Log Analytics Workspace.
  • For more, see the "Update Management" section in the Azure Monitor & Maintenance Guide.

6. Backup & Disaster Recovery (Basic Concepts for VMs)

  • Azure Backup:
    • Native service to back up Azure VMs. Uses Recovery Services Vaults.
    • Supports application-consistent snapshots for Windows, file-system consistent for Linux.
    • Restore options: entire VM, individual disks, file/folder level.
  • Azure Site Recovery (ASR):
    • For disaster recovery of VMs to another Azure region (or on-prem to Azure).
    • Replicates VMs and orchestrates failover/failback.
  • Detailed backup concepts are in the "Azure Backup" section of the Azure Monitor & Maintenance Guide. ASR is a broader topic.

AZ-104 Key Takeaways

This guide covered key aspects of Azure Virtual Machines. Here's a summary of critical points for your AZ-104 preparation:

  • IaaS Foundation: Understand VMs as Infrastructure as a Service, giving you OS control but also OS management responsibility.
  • Planning is Key:
    • Choose appropriate Regions and VM Sizes/Series (General Purpose, Compute Optimized, Memory Optimized, etc.) based on workload.
    • Understand B-series (burstable) for cost savings on variable workloads.
  • Availability:
    • Differentiate Availability Sets (protects against hardware failure in a DC) vs. Availability Zones (protects against DC failure in a region).
    • Know single VM SLA requirements (Premium SSD/Ultra Disk).
  • VM Storage:
    • OS Disks, Data Disks, and the non-persistent Temporary Disk.
    • Managed Disk types (Standard HDD/SSD, Premium SSD, Ultra Disk) and their use cases.
    • Host Caching options (ReadOnly, ReadWrite, None) and implications.
    • Understand Azure Disk Encryption (ADE) concepts.
  • OS Images:
    • Marketplace images vs. Custom Images (generalized).
    • Purpose of Azure Compute Gallery (Shared Image Gallery) for image management.
  • Deployment Methods:
    • Familiarity with Portal, Azure CLI, Azure PowerShell.
    • Understand ARM Templates for declarative, consistent deployments (IaC).
  • VM Networking:
    • NICs, Private IPs (static/dynamic), Public IPs (SKUs, static/dynamic, DNS labels).
    • Azure-provided vs. Custom DNS.
    • Role of NSGs in filtering traffic (associated with NICs/subnets).
  • Management & Operations:
    • Connecting (RDP/SSH, Azure Bastion).
    • VM states (Start, Stop (Deallocate), Restart).
    • Resizing VMs and prerequisites.
    • Basic monitoring, patching (Update Management), and backup (Azure Backup) concepts.
  • VM Scale Sets (VMSS): Understand their role for auto-scaling and managing groups of VMs.

Best Practice: Always choose the most appropriate VM size, availability option, and storage type for your workload to balance performance, cost, and resilience. Leverage automation (CLI, PowerShell, ARM Templates) for repeatable tasks. Prioritize security using NSGs, Azure Bastion, and timely patching.
