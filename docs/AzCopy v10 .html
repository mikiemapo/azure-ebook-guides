High-Performance CLI Tool

A command-line utility optimized for copying data to/from Azure Blob Storage, Azure Files, and Azure Data Lake Storage Gen2.

  • Performance: Designed for speed, using parallelism for large and numerous file transfers.
  • Scriptable: Essential for automating data transfer tasks (PowerShell, Bash).
  • Cross-Platform: Single executable for Windows, macOS, and Linux. No installation needed.
  • Supported Services:
    • Azure Blob Storage (including ADLS Gen2 hierarchical namespace)
    • Azure Files
    • *Limited support for Table Storage copy (account to account).*

Key Advantages for Data Transfer

When AzCopy is the preferred tool for Azure Storage operations.

  • Bulk Data Movement: Ideal for migrating large datasets or transferring many files.
  • Automation & Scripting: Easily integrate into DevOps pipelines or scheduled tasks.
  • Resumable Transfers: Automatically resumes interrupted copy or sync jobs from the point of failure.
  • Synchronization: Efficiently keep local directories in sync with Azure Storage, or between Azure locations.
  • Inter-Account/Region Copy: Perform server-side copies directly between storage accounts (even across regions) without downloading data locally first.

  • Azure Active Directory (Azure AD) - Recommended:
    • Use azcopy login to authenticate interactively (user) or programmatically with a Service Principal or Managed Identity.
    • Authorizes operations based on Azure RBAC roles assigned to the identity (e.g., `Storage Blob Data Contributor`, `Storage File Data SMB Share Contributor`).
    • Most secure, as credentials (like account keys or SAS tokens) are not embedded in scripts or URLs.
  • Shared Access Signature (SAS) Token:
    • Append a valid SAS token (granting specific permissions for a limited time) directly to the source and/or destination URL in AzCopy commands.
    • Useful for one-off tasks, scenarios where Azure AD login isn't practical, or for delegating limited access to third parties.
    • Example for destination: `https://[account].blob.core.windows.net/[container]?[SAS_TOKEN]`
While possible, using Storage Account Keys directly is generally discouraged for security. Prefer Azure AD or SAS.

The Workhorse: Transferring Data

Used for uploading, downloading, and copying between Azure Storage locations.

azcopy copy '<source_path_or_url>' '<destination_path_or_url>' [flags]

Common Scenarios & Flags:

  • Upload local folder to Blob:
    `azcopy copy './local-folder' 'https://[acct].blob.core.windows.net/[container]' --recursive`
  • Download Blob container to local:
    `azcopy copy 'https://[acct].blob.core.windows.net/[container]' './local-folder' --recursive`
  • Copy Blob container to another account:
    `azcopy copy 'https://[src_acct].blob.core.windows.net/[src_container]' 'https://[dest_acct].blob.core.windows.net/[dest_container]' --recursive`
  • Upload to specific Blob tier:
    `azcopy copy './file.txt' 'https://[acct].blob.core.windows.net/[container]/file.txt' --block-blob-tier Cool`
  • Filter files: --include-pattern="*.log", --exclude-path="archive/"
  • Other flags: --overwrite (control overwrite behavior), --blob-type, --log-level.

The `copy` command transfers all specified source files/blobs. If destination exists, it might be overwritten depending on flags/defaults.

Keeping Directories Aligned

Synchronizes the content of a source location to a destination location, only transferring changes.

  • Incremental & Efficient: Compares filenames and last modified timestamps (LMT). Only copies files if they are new at the source or if the LMT at the source is more recent.
  • Deletion (Optional but Powerful):
    • --delete-destination=true: Deletes files/blobs from the destination if they don't exist at the source. Irreversible!
    • --delete-destination=prompt: Prompts before deleting.
  • One-Way Sync: By default, changes flow from source to destination.
  • Ideal for keeping a local backup in Azure, or ensuring a remote site mirrors a primary location.

More AzCopy Capabilities

  • azcopy login: Authenticate using Azure AD (interactive, service principal, managed identity).
    • Use `--tenant-id` for specific tenants.
  • azcopy logout: Clear cached Azure AD tokens.
  • azcopy make (or `azcopy mk`): Create a container (Blob/ADLS Gen2) or file share. Example: `azcopy make 'https://[acct].blob.core.windows.net/newcontainer'`
  • azcopy list (or `azcopy ls`): List contents of a container, share, or local directory. Example: `azcopy ls 'https://[acct].blob.core.windows.net/mycontainer'`
  • azcopy remove (or `azcopy rm`): Delete blobs, files, or directories. Example: `azcopy rm 'https://[acct].blob.core.windows.net/mycontainer/oldfile.txt'`
  • Job Management:
    • azcopy jobs list: Show recent jobs and their status.
    • azcopy jobs show <jobID>: Details of a specific job.
    • azcopy jobs resume <jobID>: Resume a failed/paused job.
    • azcopy jobs clean [--with-status=Cancelled|Failed|Completed]: Remove old job records.
  • Automatic Resume: AzCopy automatically attempts to resume from plan files if interrupted.
  • Logging & Plan Files: Detailed logs and plan files are created in `~/.azcopy` (Linux/Mac) or `%UserPROFILE%\.azcopy` (Windows). Control log level with --log-level.
  • Performance Tuning: Environment variables like AZCOPY_CONCURRENCY_VALUE can adjust parallelism.
