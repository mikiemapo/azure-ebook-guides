This guide covers creating a Linux VM, attaching and preparing a new data disk using Azure CLI and Custom Script Extension, and then conceptually discusses enabling Azure Disk Encryption (ADE) for all disks.

A summary of the steps you performed in the Cloud Shell.

CLI: `az vm create --name support-web-vm01 --image Canonical:UbuntuServer...` (Deploy Linux VM)

`az vm create` & `az vm disk attach`

Using Azure CLI to provision a Linux VM and attach a new data disk.

# Create Linux VM
az vm create --name support-web-vm01 --image Canonical:UbuntuServer:16.04-LTS:latest --size Standard_DS1_v2 --admin-username azureuser --generate-ssh-keys

# Attach a new 64GB Premium LRS data disk
az vm disk attach --vm-name support-web-vm01 --name uploadDataDisk1 --size-gb 64 --sku Premium_LRS --new
  • A Linux VM was created using a specific Ubuntu image. SSH keys were auto-generated.
  • --new flag in `az vm disk attach` creates a new managed disk and attaches it.
  • The disk is attached but not yet usable by the OS until partitioned, formatted, and mounted.

Connecting & Verifying

You used PowerShell syntax for variable assignment in Cloud Shell, then SSH'd in.

# Note: 'az' CLI commands are run within PowerShell here
$ipaddress = az vm show --name support-web-vm01 --show-details --query "publicIps" --output tsv
Write-Output $ipaddress

ssh azureuser@$ipaddress lsblk
  • Initial `ipaddress=$(az ...)` failed because that's Bash variable assignment syntax. $ipaddress = az ... is correct for PowerShell.
  • SSH connection established successfully.
  • lsblk output initially showed the new disk (e.g., sdc) as a raw, unpartitioned device.

Automating Guest OS Configuration

Using CLI to run a script inside the Linux VM to prepare the new data disk.

az vm extension set --vm-name support-web-vm01 --name customScript \
  --publisher Microsoft.Azure.Extensions \
  --settings '{"fileUris":["https://raw.githubusercontent.com/.../add-data-disk.sh"]}' \
  --protected-settings '{"commandToExecute": "./add-data-disk.sh"}'
  • fileUris: Downloads the specified script (`add-data-disk.sh`) to the VM.
  • commandToExecute: Executes the downloaded script on the VM. This script handles:
    • Partitioning the new disk (e.g., `/dev/sdc` -> `/dev/sdc1`).
    • Creating a filesystem (e.g., ext4) on the partition.
    • Creating a mount point (e.g., `/uploads`).
    • Mounting the partition to the mount point.
    • (Typically) Adding an entry to `/etc/fstab` for persistence across reboots.
  • Subsequent `ssh ... lsblk` confirmed the disk (`sdc1`) was prepared and mounted (e.g., at `/uploads`).

Encrypting OS and Data Disks

While your lab focused on disk prep, the next step for security could be ADE. For Linux, ADE uses DM-Crypt.

  • Key Vault Required: Similar to Windows ADE, a Key Vault enabled for disk encryption is needed to store encryption keys/passphrases.
  • Cmdlets/CLI:
    • PowerShell: `Set-AzVmDiskEncryptionExtension -VolumeType All ...` (or `OS`, `Data`).
    • CLI: `az vm encryption enable --volume-type All ...`
  • Linux Mechanism: Uses DM-Crypt to encrypt the disks.
  • After adding and preparing your data disk (`/uploads`), running ADE with -VolumeType All would encrypt both the OS disk and this new data disk.
  • Status can be checked with `Get-AzVmDiskEncryptionStatus` (PS) or `az vm encryption show` (CLI).

For Linux ADE, ensure the VM image and distribution are supported. Prerequisites might include specific kernel modules or packages.
