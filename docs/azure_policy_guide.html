  • Azure Policy is a governance service that enforces organizational rules across resources.
  • Evaluates resources at creation and on schedule to ensure compliance.
  • Focus: consistency, security, and regulatory alignment.
  • Analogy: an automatic “rule enforcer” that keeps Azure tidy.
AZ-104 Focus: Understand what Azure Policy is and its governance role.

Building Blocks of Enforcement

  • Policy Definition – The rule (JSON logic + effect). Example: “Deny creation without tag ‘Environment’.”
  • Initiative Definition – Group of multiple policy definitions. Example: “Enable Security Baseline.”
  • Assignment – Applies a definition/initiative to a scope (subscription or resource group).
  • Parameters – Make policies reusable. Example: “TagName = Department.”
  • Remediation Task – Fixes non-compliant existing resources. Example: DeployIfNotExists to enable encryption.
Mnemonic: P-I-A → Definition → Initiative → Assignment.
  • 1. Create a Policy Definition (JSON rule + effect).
  • 2. Assign it to a scope (Management Group, Subscription, or Resource Group).
  • 3. Evaluate resources – new and existing.
  • 4. Enforce or flag non-compliance based on effect.
  • 5. Remediate where possible (DeployIfNotExists or Modify).

Evaluation Triggers: Resource creation, update, periodic scan.

Compliance Dashboard: Shows compliance percentage and details.

AZ-104 Focus: Know the end-to-end flow and where evaluation happens.
  • Deny – Blocks non-compliant creation/update. Example: Deny VM without tag.
  • Audit – Flags non-compliant resources but allows deployment.
  • Append – Adds required property or tag automatically.
  • Modify – Changes configuration to make resource compliant.
  • DeployIfNotExists – Creates missing resource (e.g., diagnostic settings).
Mnemonic: D-A-A-M-D → Deny, Audit, Append, Modify, DeployIfNotExists.
AZ-104 Focus: Match the correct effect to a given scenario.
  • Remediation tasks apply policy “fix” logic to existing resources.
  • Requires a Managed Identity with Contributor role.
  • Can run automatically or manually.
  • Common uses: Apply diagnostic settings, enable encryption, or deploy missing agents.
  • Exam Tip: Remediation = Governance automation, not repair or autoscale.

Policy + RBAC + Blueprints

  • RBAC defines who can do what.
  • Policy defines what must be true.
  • Blueprints bundle policies, RBAC, and templates for consistent deployments.
  • Scenario: Enforce secure settings across subscriptions using Blueprints.
AZ-104 Focus: Distinguish Policy enforcement from RBAC permissions.
  • P-I-A-R: Policy → Initiative → Assignment → Remediation.
  • D-A-A-M-D: The five Policy Effects.
  • Scope Order: Management Group → Subscription → Resource Group → Resource.
  • Compliance Dashboard: Tracks overall compliance %.
  • Always Customer Task: Interpret compliance results and act on them.
