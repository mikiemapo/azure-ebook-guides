The Basics Revisited

Remember the "Files in Folders" analogy.

  • Storage Account: Top-level namespace.
  • Container: Like a folder, organizes blobs.
  • Blob: Like a file (specifically Block Blobs for replication). Stores data objects.

Automatic History

Automatically saves previous states on modification/deletion.

  • Automatic Creation: When a blob is overwritten or deleted, the previous state becomes a distinct version (a full copy).
  • Read-Only History: Previous versions can be read or restored but not modified.
  • Account Level Setting: Enabled on the storage account.
  • Crucial Prerequisite for Object Replication.

Asynchronous Blob Copying

Copies block blobs between a source and destination storage account.

Source Account
Replication Policy
Destination Account
  • Purpose: Create a secondary copy of your data in another account (same or different region).
  • Asynchronous: Copying doesn't happen instantaneously. There's a delay (usually minutes, but depends on factors). Azure provides a replication lag SLA (99.9% within 15 mins, typically).
  • Policy-Driven: You configure replication rules on the source account specifying source/destination containers.
  • Targets Block Blobs: Specifically designed for block blobs (including their versions).

Benefits & Use Cases

Scenarios where object replication is valuable.

  • Disaster Recovery (DR): Maintain a readable secondary copy in a different region in case the primary region fails.
  • Minimize Latency: Serve read requests from a destination account geographically closer to users/compute.
  • Optimize Compute Workloads: Process data locally in different regions without impacting the source.
  • Compliance & Data Distribution: Meet requirements to store data copies in specific locations.

Must Haves Before You Start

Object replication won't work without these.

  • Account Types: Both source and destination accounts must be GPv2, Premium Block Blob, or ADLS Gen2.
  • Blob Versioning MUST be enabled on BOTH the source and destination accounts.
  • Blob Feed (Optional but Recommended): Enabling blob change feed on the source account can sometimes improve performance (dependency tracking).
  • RBAC Permissions: Appropriate roles needed on both source (to read data, configure policy) and destination (to write data) accounts.
If versioning isn't enabled on both accounts, policy creation might fail or replication won't occur correctly.

Defining the Copy Process

Configured on the source account to define what gets copied where.

  • Source & Destination: Policy specifies the destination account and region.
  • Rule Pairs: Each policy contains one or more rules defining source and destination containers.
  • Prefix Filtering (Optional): Within a rule, you can specify prefixes to only replicate blobs whose names start with that prefix (e.g., copy only `images/` from the source container).
  • Configuration: Set up via Azure Portal, PowerShell, CLI, or ARM templates.

# Conceptual Policy Structure
Policy on SourceAccount {
  DestinationAccount: "destAccount"
  Rules: [
    {
      SourceContainer: "container1"
      DestinationContainer: "backup-container1"
      Filters: { PrefixMatch: ["logs/", "archive/data"] } // Optional
    },
    {
       SourceContainer: "container2"
       DestinationContainer: "copy-container2"
       // No filters -> copy all block blobs/versions
    }
  ]
}

Version Deletion NOT Replicated
  • Versioning is Key: Replication essentially monitors the source for new versions (created by writes, overwrites, or deletes of the current blob) and copies those versions.
  • Overwrites = New Version Copy: When you overwrite a blob, the old state becomes a version, the new state is the current blob. *Both* the new current blob and the previous version exist on source and destination after replication.
  • Deletes = Delete Marker Version Copy: Deleting the *current* blob creates a new version that acts as a delete marker. This *delete marker version* is replicated. The destination account will then also show no current blob, but retains previous versions.
  • Deleting Specific Versions is NOT Replicated: If you delete an older version directly on the source, that deletion action does not replicate to the destination. The destination will keep that specific version.

Replicated:

  • New Block Blobs (creates a current version).
  • Overwrites of Block Blobs (creates a new current version, previous state becomes a version - both end up replicated).
  • Deletes of current Block Blobs (creates a delete marker version which is replicated).
  • Blob properties and metadata.
  • Blob-level immutability policies (Time-based/Legal Hold) *if configured*.

NOT Replicated:

  • Versions that existed before the replication policy was configured.
  • Deletion of specific blob versions on the source.
  • Append Blobs, Page Blobs.
  • Blob Snapshots (use Versioning instead).
  • Container-level settings (metadata, ACLs, container immutability policies - must be set independently).
  • Soft-deleted blobs (unless they are undeleted, creating a new 'current' version).
  • Blob Index Tags (must be set separately on destination if needed).

Checking Progress

How to verify replication is working.

  • Blob Properties: Check the 'Object Replication Status' property for individual blobs on the source account (Completed, Failed).
  • Policy Status: View overall policy health and rule status in the Azure portal (source account -> Object Replication).
  • Azure Monitor Metrics: Track `Blob Replication Lag Time` (preview) to monitor the delay between source write and destination completion.
  • If status is 'Failed', check permissions, ensure versioning is enabled on both accounts, and verify container existence.

Important Notes

  • Cost: Incur costs for:
    • Replication Transactions (per blob/version copied).
    • Data Egress (if replicating cross-region).
    • Storage costs on the destination account (including all replicated versions).
  • Asynchronous Nature: Not suitable for scenarios requiring zero data loss (RPO=0) on failover, as there will always be some replication lag.
  • Immutability: Replication respects immutability on the destination. If a destination blob is immutable, replication of an overwrite/delete marker might fail until the policy allows it.
  • Lifecycle Management: Can be configured independently on source and destination accounts. You might want different policies (e.g., keep versions longer on destination DR account).
  • One-Way: Replication is configured source -> destination. It doesn't automatically replicate changes made on the destination back to the source.
