Containers & Blobs

The fundamental structure.

  • Container: A logical grouping for blobs, like a folder. Forms part of the blob's URL.
  • Blob (Block Blob): The object storing your data (files, images, etc.).

Keeping Previous States

The core idea behind any versioning system.

Document v1 (Previous)
  • Preserving History: Allows you to access or restore older copies of a file or object.
  • Protection: Safeguards against accidental modification or deletion by keeping prior copies accessible.
  • Rollback Capability: Enables reverting to a known good state.

Automatic State Capture

Azure's built-in mechanism for saving previous blob states.

Blob v1 (Current)
Overwrite / Delete
Version 1 (Previous)
Blob v2 (Current) / (None if Deleted)
  • Enabled at Account Level: A setting on the storage account.
  • Triggers: Creates a new version automatically on overwrite or deletion of the current blob.
  • Immutable Versions: Versions are read-only snapshots.
  • Version ID: Each version gets a timestamp ID; the current blob does not.
  • Cost Implication: Each version is a full blob and incurs storage costs.

Automated Tiering & Deletion

Rule-based automation for managing blob lifespan and cost.

Rule:
Filter + Action
Lifecycle Engine (Daily Run)
Tier Blob / Version (Hot/Cool/Cold/Archive)
OR
Delete Blob / Version / Snapshot
  • Policy & Rules: Define rules with filters (prefix, tags) and actions.
  • Actions: `TierToCool`, `TierToCold`, `TierToArchive`, `Delete`.
  • Conditions: Based on age (`daysAfterModificationGreaterThan`, `daysAfterCreationGreaterThan`, `daysAfterVersionCreation`) or last access time (`daysAfterLastAccessTimeGreaterThan`).
  • Primary Goal: Reduce storage costs by moving old data to cheaper tiers and removing obsolete data.

Managing Versions Automatically

Lifecycle rules can specifically target blob versions to manage costs and retention.

Blob Version Created (e.g., 100 days ago)
Lifecycle Rule: `IF subtype == 'blobVersion' AND daysAfterVersionCreation > 90 THEN TierToCool`
Version automatically moved to Cool tier after 90 days.
Lifecycle Rule: `IF subtype == 'blobVersion' AND daysAfterVersionCreation > 365 THEN Delete`
Version automatically deleted after 365 days.
  • Targeting Versions: Lifecycle rules use `daysAfterVersionCreation` condition and/or filter for `blobVersion` subtype to specifically manage previous versions.
  • Tiering Versions: Move older, less frequently accessed versions to Cool, Cold, or Archive tiers to save costs.
  • Deleting Versions: Automatically clean up old versions after a defined retention period (e.g., delete versions older than 1 year).
  • Crucial for Cost Management: Since every version incurs cost, using Lifecycle Management to tier or delete old versions is essential to prevent unbounded cost growth.
  • Base Blob vs. Version Rules: You can have separate rules for the current (base) blob (using `daysAfterModification` / `daysAfterLastAccess`) and its previous versions (using `daysAfterVersionCreation`).
Without Lifecycle Management rules targeting versions, old versions will persist indefinitely in their original tier, potentially leading to high storage costs.

Key Takeaways

  • Versioning is enabled at the account level.
  • Versioning automatically protects against overwrites/deletes by creating read-only previous states.
  • Versions incur storage costs.
  • Lifecycle Management automates tiering/deletion based on rules.
  • Lifecycle rules can and should be used to manage the lifespan and cost of Blob Versions.
