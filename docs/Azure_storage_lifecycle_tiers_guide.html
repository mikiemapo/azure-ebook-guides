  • Premium (Block Blobs): Lowest latency (SSD), highest cost. For interactive workloads, high transaction rates. *Not a lifecycle target tier.*
  • Hot: Optimized for frequent access. Higher storage cost, lower access cost. Default tier.
  • Cool: For infrequent access (min 30 days). Lower storage cost, higher access cost than Hot.
  • Cold: For rare access (min 90 days). Even lower storage cost, higher access cost than Cool.
  • Archive: For long-term retention (min 180 days), rarely accessed. Lowest storage cost, highest access cost. High retrieval latency (hours). Offline tier.

Lifecycle management automates moving blobs between Hot, Cool, Cold, and Archive tiers.

Automating Storage Tiering & Deletion

Define rules to automatically transition blobs to cooler tiers or delete them based on conditions.

  • Rule-Based Engine: You define policies with rules containing filters and actions.
  • Cost Optimization: Primary goal is to move less frequently accessed data to cheaper tiers.
  • Data Retention: Automatically delete blobs, versions, or snapshots after a certain period.
  • Applies to: Block blobs and Append blobs within specific storage account types.

Filters + Actions

Each rule within a policy consists of a filter set and an action set.

Lifecycle Rule
  • Filter Set: Defines which blobs the rule applies to (e.g., based on prefix, tags, blob type).
  • Action Set: Defines what happens to the filtered blobs when the condition (e.g., age) is met.
  • A policy can contain multiple rules (up to 100 per storage account).
  • Rules are defined in JSON format within the storage account's properties.

Selecting Blobs for Actions

Conditions used to target specific blobs.

  • Prefix Match: Apply rule to blobs starting with a specific path (e.g., `container1/logs/`).
  • Blob Index Tags: Apply rule based on key-value index tags associated with blobs (e.g., `Project = 'Blue'`).
  • Blob Type: Specify `blockBlob` or `appendBlob`.
  • Blob Subtype: Target base blobs, snapshots (`blobSnapshot`), or versions (`blobVersion`).

Filters define the scope of a rule's actions.

Tiering and Deleting Based on Age

The operations performed when conditions are met.

Common Actions:

  • TierToCool: Move blob to Cool tier.
  • TierToCold: Move blob to Cold tier.
  • TierToArchive: Move blob to Archive tier.
  • Delete: Delete the base blob.
  • DeleteBlobSnapshot: Delete previous blob snapshots.
  • DeleteBlobVersion: Delete previous blob versions.

Common Conditions (Age in days):

  • daysAfterModificationGreaterThan: Based on last write time.
  • daysAfterCreationGreaterThan: Based on blob creation time.
  • daysAfterLastAccessTimeGreaterThan: Requires access tracking to be enabled. Based on last read/write.

Combine filters and actions/conditions to build your policy (e.g., "If blob prefix is 'archive/' AND last modified > 90 days, THEN TierToArchive").

Enabling Last Access Time tracking incurs a small cost but provides more accurate tiering for read-heavy workloads.

How and When Rules Run

Lifecycle policies run automatically in the background.

  • Frequency: Rules run once per day.
  • Timing: There's no SLA for the exact time rules run. It can take up to 24 hours for a rule run to complete across the entire account.
  • Policy Updates: If you update a policy, it can take up to 24 hours for the changes to take effect.
  • Idempotent: If a rule fails (e.g., transient issue), it will retry on subsequent runs.

Account & Blob Types

Where lifecycle management policies can be applied.

  • Account Types:
    • General-purpose v2 (GPv2)
    • BlobStorage (legacy, block blobs only)
    • Premium Block Blob
  • Blob Types:
    • Block Blobs (Most common)
    • Append Blobs
  • Not Supported: General-purpose v1 (GPv1), Page Blobs, Azure Files shares, Premium Page Blobs, Premium File Shares.

Important Notes

  • Pricing:
    • Executing rules: Small cost per 10,000 rules executed.
    • Tier changes: Per-transaction costs apply (Set Blob Tier).
    • Access Tracking: Cost per 10,000 transactions if enabled.
    • Early Deletion Fees: Apply for Cool, Cold, and Archive tiers if blobs deleted before minimum retention (30/90/180 days). Lifecycle deletion avoids this if age condition matches minimum.
  • Last Access Time Tracking: Must be explicitly enabled on the storage account if using `daysAfterLastAccessTimeGreaterThan`.
  • Immutability Interaction: Lifecycle management cannot delete or transition an immutable blob until its policy expires / holds are cleared.
  • Soft Delete Interaction: Lifecycle deletion respects soft delete retention periods. The blob enters the soft-deleted state.
