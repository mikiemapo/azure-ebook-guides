Files and Folders in the Cloud

Think of Azure Blob Storage like a massive file system.

Your Computer
Folder (e.g., 'MyDocs')
File (e.g., 'report.docx')
Azure Storage Account
Container
Blob (Binary Large Object)
  • Storage Account: The top-level resource providing a unique namespace for your data.
  • Container: Like a folder. Organizes blobs and forms part of the blob's URL.
  • Blob: Like a file. Stores the actual data (images, videos, logs, documents, backups - anything!).
  • Blob Types:
    • Block Blobs: Most common; optimized for streaming and storing text/binary data (up to ~190 TiB).
    • Append Blobs: Optimized for append operations (like logs).
    • Page Blobs: Used for Azure VM disks (random read/write). *Less relevant for general blob/lifecycle concepts.*

  • Hierarchy: Storage Account -> Container -> Blob.
  • Unique URL: Every blob has a unique address: `https://.blob.core.windows.net//`.
  • Virtual Folders: You can use `/` in blob names to simulate a folder structure (e.g., `images/archive/photo.jpg`), but containers are the only true folder level.

Automatic "Backups" on Change

Preserves previous states of your blobs automatically when they are modified or deleted.

Blob v1 (Current)
Modify / Overwrite
Blob v1 (Previous Version)
Blob v2 (Current)
Blob v2 (Current)
Delete Current Blob
Blob v2 (Previous Version, marked for deletion if Soft Delete on)
No Current Blob
  • Enabled at Storage Account Level: Not per container or blob.
  • Automatic: When a blob is overwritten or deleted, the previous state becomes a version automatically.
  • History: Provides a history of changes, allowing you to view or restore previous states.
  • Unique ID: Each version has a unique Version ID (timestamp). The "current" blob doesn't have a version ID in its main URL.
  • Immutable Versions: Version content cannot be changed. Immutability policies can be applied per-version.
  • Acts like a backup for previous states but isn't a managed backup service (no central restore portal etc.).

  • Rule-Based: Uses rules with filters (prefixes, tags) and actions (tier, delete).
  • Purpose: Primarily cost optimization by moving old/unused data to cheaper tiers and cleaning up obsolete data.
  • Can Target: Rules can be configured to act on base blobs, previous versions, or snapshots based on their age.

Working Together (Key AZ-104 Area!)

How these two features cooperate to manage data states and retention.

Versioning ON, Blob Modified
Old state becomes a version. New state is current blob.
Lifecycle Rule: Delete 'Previous Versions' > 90 days old
Versions older than 90 days are deleted. Current blob remains.
Lifecycle Rule: Delete 'Base Blob' > 365 days old
Current blob becomes a version (marked deleted if Soft Delete on). No current blob exists.
+ Lifecycle tries to Delete Immutable Version
Action fails until immutability expires / hold removed.
  • Targeting Versions: Lifecycle rules can specifically target previous versions using the `daysAfterVersionCreation` condition or by filtering for `blobVersion` subtype.
  • Cleaning Old Versions: Common use case is to delete versions after a certain period (e.g., delete versions > 90 days old).
  • Deleting Base Blobs with Versioning: If lifecycle deletes the base blob when versioning is enabled, the current state becomes a *new version*. The base blob itself ceases to exist.
  • Order Matters: Be careful setting rules. Deleting base blobs too early might leave you only with versions. Deleting versions too early might remove desired history.

Soft Delete & Immutability

Additional layers that interact with Versioning and Lifecycle.

  • Soft Delete: Retains deleted blobs/versions for a configured period, allowing recovery. Lifecycle deletion respects this; items enter soft-deleted state first.
  • Immutability: Prevents modification/deletion for a time or via legal hold. Overrides Lifecycle actions and Soft Delete until the policy/hold is no longer active.
  • Combined Power: Using Versioning + Soft Delete + Immutability + Lifecycle provides comprehensive data protection and management.
