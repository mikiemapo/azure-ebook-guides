Server-Side Asynchronous Copy

An operation to copy a single blob to a destination within the same or a different storage account.

  • Operation Type: It's an API call (or SDK function / CLI command) you initiate.
  • Server-Side: Once initiated, Azure performs the copy internally between the source and destination. Your client doesn't need to stay connected or download/upload the data.
  • Asynchronous: The copy operation starts in the background. The initial request returns quickly, but the actual data transfer takes time. You need to check the status to know when it's complete.
  • Individual Blob Focus: Operates on one blob at a time. To copy multiple blobs, you need to initiate multiple copy operations (often via scripting).
This is different from Object Replication, which is a policy-based, continuous replication service requiring versioning. `Copy Blob` is a one-time, per-blob copy request.

  1. Client (e.g., script, application, tool) sends a `Copy Blob` request, specifying the source blob URL and the destination account, container, and blob name.
  2. Azure Storage validates the request and queues the copy operation. It returns a response immediately with a copy ID and initial status (usually `pending`).
  3. Azure performs the data transfer server-side in the background.
  4. Client must periodically check the copy status using the destination blob name or the copy ID (e.g., via `Get Blob Properties`) until the status is `success` or `failed`.

When to Use `Copy Blob`

  • Creating Manual Backups: Copying important blobs to a separate container or account.
  • Moving Blobs: Copy a blob to a new location and then delete the original (effectively a move).
  • Aggregating Data: Copying logs or results from multiple sources into a central location.
  • Changing Tiers During Copy: You can specify a different access tier for the destination blob during the copy initiation (e.g., copy from Hot to Archive).
  • Hydrating Archive Blobs: Initiate a copy from an Archive blob to a Hot/Cool/Cold destination to rehydrate it (copy operation handles rehydration priority).

Important Characteristics

  • Asynchronous: Completion time varies based on size, load, source/destination regions. Don't assume immediate availability.
  • Server-Side: Frees up client resources; Azure does the heavy lifting.
  • Source/Destination Flexibility: Can copy within the same account, between accounts in the same region, or between accounts in different regions.
  • Authentication: Needs appropriate permissions (SAS token, RBAC, Account Key - depending on scenario) for both source read and destination write.
  • Overwriting: By default, `Copy Blob` overwrites the destination blob if it exists. If versioning is enabled on the destination, the overwrite creates a new version.
  • Tiering Options: Can specify destination tier (`x-ms-access-tier` header) and rehydration priority (`x-ms-rehydrate-priority` header) for Archive source.

Monitoring Progress

Crucial due to the asynchronous nature.

  • Initiation: Returns HTTP status `202 Accepted` and copy information in headers.
  • Checking Method: Use `Get Blob Properties` on the destination blob.
  • Status `pending`: Copy is in progress. `Copy Progress` header shows `bytes_copied/total_bytes`.
  • Status `success`: Copy completed successfully.
  • Status `failed` / `aborted`: Copy did not complete. Check `Copy Status Description` for error details.
  • Abort Copy Blob: You can attempt to cancel a pending copy operation.

Important Notes

  • Latency: Factor in the asynchronous delay, especially for large blobs or cross-region copies.
  • Cost: Incur costs for:
    • Read operations on the source.
    • Write operations on the destination.
    • Data Egress (if copying cross-region).
    • Storage costs for the new blob at the destination.
  • Versioning Interaction: If versioning is enabled on destination, overwriting creates a version. `Copy Blob` copies only the *specified* source blob/version, not its entire history.
  • vs. Object Replication: `Copy Blob` is manual, per-blob, one-time async. Object Replication is automatic, policy-based, continuous async (for new versions), and requires versioning enabled everywhere.
  • Archive Tier Source: Copying *from* Archive requires rehydration first (unless using Copy Blob to rehydrate to Hot/Cool/Cold). Copying *to* Archive is possible directly.
