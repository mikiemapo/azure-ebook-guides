Choosing the Right Tier for Blob Data

Balance storage cost, access cost, and retrieval latency based on how often data is used.

Tier Storage Cost Access Latency / Cost
Hot
Highest Storage Cost Lowest Access Latency & Cost
Cool
Lower Storage Cost Low Latency / Higher Access Cost
Cold
Lower Storage Cost Low Latency / Higher Access Cost
Archive
Lowest Storage Cost Highest Latency (Hours) & Access Cost
  • Hot: Optimized for frequent access. Default tier. Highest storage costs, lowest access costs.
  • Cool: For infrequent access (data stored > 30 days). Lower storage costs, higher access costs than Hot. Minimum retention applies.
  • Cold: For rare access (data stored > 90 days). Lower storage costs, higher access costs than Cool. Minimum retention applies.
  • Archive: For long-term retention (> 180 days), rarely accessed. Lowest storage costs, highest access & retrieval costs. Offline, retrieval takes hours (rehydration). Minimum retention applies.
Premium Tier Clarification: Premium performance tier exists for **Premium Block Blob** accounts (using SSDs). It offers the lowest latency for high-transaction workloads. Lifecycle Management *can* run on Premium accounts to move data *out* of Premium to Hot/Cool/Cold/Archive, but Premium itself is **not a target tier** for lifecycle rules to move data *into*.

Lifecycle management automates moving blobs between Hot, Cool, Cold, and Archive tiers.

Automating Storage Tiering & Deletion

Define rules to automatically transition blobs to appropriate tiers (Hot, Cool, Cold, Archive) or delete them based on conditions.

  • Rule-Based Engine: Policies contain rules with filters and actions.
  • Cost Optimization: Move less frequently accessed data to cheaper tiers (Cool, Cold, Archive).
  • Data Retention: Automatically delete blobs, versions, or snapshots after their useful life.
  • Applies to: Block blobs and Append blobs in GPv2, BlobStorage, and Premium Block Blob accounts.

Tiering and Deleting Based on Age

The operations performed when conditions are met.

Common Actions:

  • TierToCool: Move blob to Cool tier.
  • TierToCold: Move blob to Cold tier.
  • TierToArchive: Move blob to Archive tier.
  • Delete: Delete the base blob / snapshot / version (depending on filter).

Common Conditions (Age in days):

  • daysAfterModificationGreaterThan: Based on last write time.
  • daysAfterCreationGreaterThan: Based on blob creation time.
  • daysAfterLastAccessTimeGreaterThan: Requires access tracking enabled. Based on last read/write.
  • daysAfterLastTierChangeGreaterThan: (Newer condition) Based on when the blob was last tiered.

Combine filters and actions/conditions (e.g., "If blob prefix is 'logs/' AND last accessed > 30 days, THEN TierToCool").

Enabling Last Access Time tracking incurs a small cost but provides more accurate tiering for read-heavy workloads.

Filters + Actions

Each rule within a policy consists of a filter set and an action set.

  • Filter Set: Defines which blobs the rule applies to (e.g., based on prefix, tags, blob type).
  • Action Set: Defines what happens to the filtered blobs when the condition (e.g., age) is met.
  • A policy can contain multiple rules (up to 100 per storage account).
  • Rules are defined in JSON format within the storage account's properties.

Selecting Blobs for Actions

Conditions used to target specific blobs.

  • Prefix Match: Apply rule to blobs starting with a specific path (e.g., `container1/logs/`).
  • Blob Index Tags: Apply rule based on key-value index tags associated with blobs (e.g., `Project = 'Blue'`).
  • Blob Type: Specify `blockBlob` or `appendBlob`.
  • Blob Subtype: Target base blobs, snapshots (`blobSnapshot`), or versions (`blobVersion`).

Filters define the scope of a rule's actions.

How and When Rules Run

Lifecycle policies run automatically in the background.

  • Frequency: Rules run once per day.
  • Timing: No SLA for exact start time; can take up to 24 hours to complete.
  • Policy Updates: Can take up to 24 hours for changes to apply.
  • Idempotent: Rules retry automatically if transient issues occur.

Account & Blob Types

Where lifecycle management policies can be applied.

  • Account Types: GPv2, BlobStorage, Premium Block Blob.
  • Blob Types: Block Blobs, Append Blobs.
  • Not Supported: GPv1, Page Blobs, Azure Files, etc.

Important Notes

  • Pricing: Rules execution cost, tier change transaction costs, potential early deletion fees (Cool/Cold/Archive), access tracking cost (if enabled).
  • Last Access Time Tracking: Must be explicitly enabled if using that condition.
  • Immutability Interaction: Cannot delete/transition immutable blobs prematurely.
  • Soft Delete Interaction: Lifecycle respects soft delete retention.
  • Minimum Retention: Cool (30 days), Cold (90 days), Archive (180 days). Early deletion/tiering might incur fees if done manually *before* these periods. Lifecycle actions generally align with these minimums.
