Microsoft Entra ID Core Concepts

Understanding identities, access management, and core features of Microsoft Entra ID (formerly Azure AD). Essential for AZ-104. Click/Tap headers to explore.

Cloud Identity & Access Management

Microsoft Entra ID is a cloud-based identity and access management (IAM) service provided by Microsoft. It helps organizations manage and secure identities for users, applications, and devices.

Formerly Known As: You may also see this service referred to by its previous name, Azure Active Directory (Azure AD).

Core Purpose:

  • Enable users to sign in securely.
  • Control access to resources based on organizational policies.

Resources Accessed via Entra ID:

  • External SaaS apps (e.g., Microsoft 365, Salesforce, Dropbox).
  • Internal resources (e.g., custom line-of-business cloud applications).
  • Azure portal and Azure resources (via Azure RBAC).
  • Windows devices (Microsoft Entra join/hybrid join).

Key Functions:

  • Identity Management: Creating and managing users, groups, and service principals.
  • Authentication: Verifying the identity of users and devices (e.g., username/password, MFA).
  • Authorization: Granting or denying access to applications and resources.
  • Single Sign-On (SSO): Allowing users to sign in once and access multiple applications.

AZ-104 Focus: Understand that Entra ID is the core identity service for Azure and Microsoft cloud services. It's crucial for managing users who access Azure resources.

Cloud IAM vs. On-Premises Directory

It's crucial to understand that Microsoft Entra ID and traditional Active Directory Domain Services (AD DS) serve different primary purposes, though they can work together in hybrid scenarios.

Microsoft Entra ID (Cloud-Native IAM)

  • Primary Purpose: Identity and Access Management for cloud applications (SaaS, custom cloud apps) and Azure resources.
  • Scope: Internet-facing, designed for modern web protocols.
  • Authentication Protocols: OAuth 2.0, OpenID Connect (OIDC), SAML.
  • Structure: Flat structure of users and groups within a tenant.
  • Device Join: Microsoft Entra joined, Microsoft Entra hybrid joined (for cloud management of domain-joined devices).
  • Not a direct replacement for AD DS for authenticating on-premises servers or legacy apps using Kerberos/LDAP.

Active Directory Domain Services (AD DS - On-Premises)

  • Primary Purpose: Directory services for on-premises networks. Manages users, computers, groups, GPOs.
  • Scope: Primarily internal corporate networks.
  • Authentication Protocols: Kerberos, NTLM, LDAP.
  • Structure: Hierarchical (Domains, Trees, Forests, OUs).
  • Device Join: Traditional domain join for Windows clients and servers.
  • Often extended to Azure by running AD DS domain controllers on Azure VMs.

Quick Comparison:

Feature Microsoft Entra ID AD DS (On-Premises)
Primary Use Cloud applications & Azure resources IAM On-premises network IAM, GPOs
Key Protocols OAuth 2.0, OIDC, SAML Kerberos, LDAP, NTLM
Structure Flat (Tenant) Hierarchical (Forests, Domains, OUs)
Device Management Entra join, Hybrid join, MDM/MAM focus Traditional Domain Join, Group Policy

Hybrid Identity: Microsoft Entra Connect is a tool to synchronize identities from on-premises AD DS to Microsoft Entra ID, enabling users to use the same credentials for both on-prem and cloud resources (SSO).

AZ-104 Focus: Know the distinct purposes. Entra ID is central to Azure identity. Understand that AD DS on Azure VMs is for extending on-prem AD, not a cloud-native replacement by Entra ID for traditional server auth.

User Identities in Entra ID

User accounts are fundamental to controlling access. Entra ID supports different types of user identities.

User Identity Types:

  • Cloud-Only (Cloud-Native): Users created directly in Microsoft Entra ID. Live only in the cloud. Password managed in Entra ID.
  • Synchronized: Users mastered in on-premises AD DS and synchronized to Microsoft Entra ID using Entra Connect. Can use same sign-on (SSO) methods like Password Hash Sync (PHS) or Pass-Through Authentication (PTA).
  • Guest Users (B2B): External users (e.g., partners, vendors) invited into your Entra ID tenant to collaborate or access specific resources.

Creating & Managing Users:

  • Creation Methods:
    • Azure portal / Microsoft Entra admin center.
    • Azure CLI (az ad user create).
    • Azure PowerShell (New-MgUser - Microsoft Graph PowerShell).
    • Microsoft Graph API.
  • User Principal Name (UPN): Primary login name, typically username@yourdomain.com.
  • Password Management: Resetting passwords, setting temporary passwords, self-service password reset (SSPR) options.
  • Assigning Roles: Assign built-in Entra ID roles (e.g., Global Administrator, User Administrator) or custom roles for administrative tasks. (Azure RBAC roles are separate, for Azure resource access).
  • Bulk Operations: Can create, invite, or delete users in bulk using CSV files (portal) or scripting.

AZ-104 Focus: Know how to create users, basic user management tasks (reset password), and differentiate user types (cloud, sync, guest). Understand the concept of UPN.

Organizing Users & Assigning Access

Groups in Microsoft Entra ID are used to manage collections of users (or other principals) to simplify access management and role assignments.

Group Types:

  • Security Groups:
    • Primary use is to manage access to resources (e.g., applications, Azure resources via RBAC, SharePoint sites). Can contain users, devices, other groups, and service principals.
  • Microsoft 365 Groups:
    • Enable collaboration by providing members access to shared resources like a SharePoint team site, Exchange mailbox, Planner, OneNote. Always security-enabled. Can have dynamic membership.

Membership Types:

  • Assigned: Members are manually added and removed. Most common type.
  • Dynamic User: Membership is automatically managed based on user attributes (e.g., department, location). Requires Microsoft Entra ID P1 or P2 license.
  • Dynamic Device: Membership is automatically managed based on device attributes (e.g., OS type, device model). Requires Microsoft Entra ID P1 or P2 license.

Key Uses of Groups:

  • Assigning access to applications.
  • Assigning Entra ID roles or Azure RBAC roles.
  • Managing licenses (group-based licensing). Requires P1/P2.
  • Targeting for Conditional Access policies or Intune policies.

AZ-104 Focus: Differentiate Security vs. M365 groups. Understand assigned vs. dynamic membership concepts. Know how groups are used for access assignments.

Your Organization's Entra ID Instance

Understanding the concept of a tenant and how to use custom domains for branding and identity.

Microsoft Entra Tenant:

  • A dedicated and trusted instance of Microsoft Entra ID, automatically created when your organization signs up for a Microsoft cloud service subscription (e.g., Azure, Microsoft 365).
  • Represents your organization and acts as a container for users, groups, apps, and devices.
  • Globally unique, with an initial domain name like yourtenantname.onmicrosoft.com.

Custom Domains:

  • You can add your organization's custom domain names (e.g., contoso.com) to your Entra ID tenant.
  • Allows users to sign in with UPNs that match your corporate branding (e.g., user@contoso.com).
  • Verification Process: To add a custom domain, you must prove ownership by adding a specific DNS record (TXT or MX) to your domain's public DNS zone.
  • One custom domain can be set as the primary domain for new user creation.

AZ-104 Focus: Know what a tenant is. Understand the purpose of custom domains and the high-level verification process (DNS record).

Adding Layers of Security

MFA is a security process requiring users to provide more than one form of verification to prove their identity during sign-in.

Why MFA is Critical:

  • Passwords can be compromised (phishing, breaches, weak passwords).
  • MFA provides significantly stronger protection against unauthorized access, even if a password is stolen. Microsoft states it blocks over 99.9% of identity compromise attacks.

Common MFA Factors (Something you know, have, or are):

  • Microsoft Authenticator app: Push notification approval or one-time passcode (OTP). Recommended method.
  • SMS: Code sent via text message.
  • Phone Call: Automated voice call.
  • Biometrics: Windows Hello for Business, FIDO2 security keys.
  • Hardware/Software OATH Tokens: Generate OTPs.

Enabling MFA in Microsoft Entra ID:

  • Security Defaults: Basic MFA enforcement for all users, especially admins. Enabled by default in newer tenants.
  • Conditional Access Policies: Preferred and most flexible method. Allows granular control to require MFA based on conditions (user, location, device, app, risk). Requires Entra ID P1/P2.
  • Per-user MFA (legacy method): Manually enable/enforce MFA for individual users. Not recommended for broad deployment.

AZ-104 Focus: Understand what MFA is, why it's important, common methods, and that Conditional Access is the primary way to enforce it strategically.

If-Then Access Control

Conditional Access policies in Microsoft Entra ID are "if-then" statements that bring signals together, to make decisions, and enforce organizational access policies.

Requires Microsoft Entra ID P1 or P2 licenses.

Signals (IF) User/Group, Location, Device, Application, Real-time Risk
Policy Engine
Decisions (THEN) Block, Grant (e.g., Require MFA, Compliant Device, Limited Access)

Common Signals (Conditions - "IF"):

  • User or group membership: Target specific users or groups.
  • IP Location: Based on public IP address ranges (e.g., trusted corporate network, specific countries).
  • Device: Device platform (Windows, iOS, Android), device compliance state (via Intune).
  • Application: Specific cloud app being accessed.
  • Real-time and calculated risk detection: (Requires Entra ID P2 for Identity Protection features) e.g., sign-in from anonymous IP, atypical travel.

Common Decisions (Access Controls - "THEN"):

  • Block access: Highest restriction.
  • Grant access:
    • But require MFA.
    • But require device to be marked as compliant.
    • But require Microsoft Entra hybrid joined device.
    • But require approved client app.
    • But require app protection policy (for mobile apps).
    • (Session controls like limited experience can also apply).

AZ-104 Focus: Understand the concept of Conditional Access (if-then), common signals and decisions, and its role as the primary tool for enforcing MFA and other access requirements.

Governance vs Control:

  • Governance: Policies, Blueprints, and Locks that enforce compliance and prevent misconfigurations.
  • Access Control: Entra, RBAC, and NTFS roles that decide who can take which actions.

AZ-104 Focus: Distinguish the three layers. Entra = people. RBAC = Azure resources. NTFS = file-level permissions. Understand how governance overlays them with rules and policies.

Key Takeaway:

For the AZ-104, know that permissions flow downward: Entra defines identities ‚Üí RBAC grants resource access ‚Üí NTFS defines data access. Governance tools like Azure Policy set the guardrails that keep all layers compliant.

Essential Concepts for the Exam

  • Microsoft Entra ID: Core cloud identity service (formerly Azure AD). Key roles: Global Administrator, User Administrator, Security Administrator, Guest Inviter.
  • Azure RBAC: Controls access to resources. Key roles: Owner, Contributor, Reader, Storage Account Contributor, Storage Blob Data Contributor, SMB Contributor.
  • NTFS / ACLs: File-level permissions inside resources. Key permissions: Full Control, Modify, Read & Execute, Read, Write.
  • Manage users (cloud, synchronized, guest) and groups (Security, M365; assigned, dynamic).
  • Understand tenants and custom domains.
  • Implement MFA with methods such as Microsoft Authenticator, SMS, Phone Call, Hardware Tokens, or Biometrics.
  • Use Conditional Access policies to enforce security based on signals and conditions (user, device, location, app, risk).
  • Leverage Microsoft Entra Connect for hybrid identity synchronization between on-premises AD DS and Entra ID.

For AZ-104, focus on practical administration tasks related to users, groups, and understanding how MFA and Conditional Access contribute to securing identities and access to Azure resources and cloud applications.
