Infrastructure as Code for Azure

Define the Azure resources you need in a declarative JSON file.

  • Declarative Syntax: You define what you want to deploy, not the sequence of commands to create it.
  • JSON Format: Templates are JavaScript Object Notation (JSON) files.
  • Idempotent: Deploying the same template multiple times results in the same state. Azure Resource Manager handles creating, updating, or deleting resources to match the template.
  • Orchestration: Resource Manager orchestrates the deployment, handling dependencies and parallel deployments where possible.

Think of it as a blueprint for your Azure environment.

Benefits of IaC

ARM Templates offer significant advantages for managing Azure resources.

  • Consistency & Repeatability: Ensure environments are deployed identically every time, reducing errors.
  • Orchestration: Handles complex deployments involving multiple interdependent resources automatically.
  • Modularity: Break templates into smaller, reusable components using linked or nested templates.
  • CI/CD Integration: Automate infrastructure deployment as part of your DevOps pipeline.
  • Exportable Code: Export the template for existing resource groups to understand the JSON or reuse it.
  • Authoring Tools: Good tooling support (VS Code extension, Azure Portal).
  • Testability: Validate templates before deployment to catch errors early.

File Format / Sections

ARM templates follow a specific JSON structure.

"$schema": Location of the JSON schema file (defines structure & validation). Usually points to a Microsoft schema.
"contentVersion": Version of your template (e.g., "1.0.0.0"). Required.
"apiProfile": (Optional) API version profile for Azure Stack environments.
"parameters": (Optional) Values provided during deployment to customize resources (e.g., VM name, size).
"variables": (Optional) Values constructed and used within the template for simplification (e.g., concatenating names).
"functions": (Optional) User-defined functions to reuse complex expressions.
"resources": Core section defining the Azure resources to deploy (VMs, storage, networks, etc.). Required.
"outputs": (Optional) Values returned after deployment (e.g., a public IP address, connection string). 

{
  "$schema": "https://schema.management.azure.com/schemas/2019-04-01/deploymentTemplate.json#",
  "contentVersion": "1.0.0.0",
  "parameters": { ... },
  "variables": { ... },
  "functions": [ ... ],
  "resources": [ ... ],
  "outputs": { ... }
}

Inside the Template

Understanding the main sections.

  • Parameters: Define inputs needed at deployment time. Can have types (string, int, bool, etc.), default values, allowed values, descriptions.
  • Variables: Define values used internally. Useful for complex expressions, resource naming conventions, reducing repetition. Not meant for input.
  • Functions: Built-in functions (e.g., `concat()`, `resourceGroup()`, `uniqueString()`) and user-defined functions simplify expressions.
  • Resources: An array of resource objects. Each object specifies `type`, `apiVersion`, `name`, `location`, `properties`, and potentially `dependsOn` for ordering.
  • Outputs: Define values to return after deployment. Useful for chaining deployments or retrieving connection details.
  • Expressions: Use `[function()]` syntax within JSON strings to dynamically generate values.

Where You Can Deploy

Templates can target different levels of the Azure hierarchy.

  • Tenant: Deploy resources like management groups or Azure AD policy assignments.
  • Management Group: Deploy policies or role assignments across multiple subscriptions.
  • Subscription: Deploy resource groups, policies, role assignments at the subscription level.
  • Resource Group: Most common scope. Deploy resources like VMs, storage accounts, VNets within a specific resource group.

The `$schema` and API versions used might differ slightly depending on the scope.

Creating Templates

Tools to help you write and manage ARM templates.

  • Visual Studio Code (VS Code): Recommended. Use the Azure Resource Manager (ARM) Tools extension for intellisense, validation, snippets, and schema integration.
  • Azure Portal: Export templates from existing resources or resource groups. Deploy templates directly. Limited authoring experience.
  • Visual Studio: Provides Azure Resource Group project type with template authoring support.
  • Plain Text Editor: Possible, but lacks validation and intellisense, making it error-prone.

Running Your Templates

Various ways to deploy your infrastructure using ARM templates.

  • Azure Portal: Upload and deploy templates via the "Deploy a custom template" service.
  • Azure CLI: Use `az deployment group create` (for RG scope), `az deployment sub create`, etc.
  • Azure PowerShell: Use `New-AzResourceGroupDeployment` (for RG scope), `New-AzSubscriptionDeployment`, etc.
  • Azure SDKs: Programmatically deploy templates using languages like Python, .NET, Java, etc.
  • CI/CD Pipelines: Integrate deployment into Azure DevOps Pipelines, GitHub Actions, Jenkins, etc.
  • Azure Cloud Shell: Access CLI and PowerShell directly in the browser.

A DSL for ARM

An abstraction layer on top of ARM Templates for a simpler authoring experience.

  • Domain-Specific Language (DSL): Provides a cleaner, more concise syntax compared to JSON.
  • Transpiles to ARM JSON: Bicep files are compiled into standard ARM Template JSON before deployment. All ARM capabilities are available.
  • Improved Type Safety & Validation: Better tooling support for catching errors during authoring.
  • Modularity: Easier module creation and referencing.

If you find ARM JSON verbose, Bicep is often preferred for new development.

Storing Templates in Azure

A way to store ARM templates within Azure for easier sharing and reuse.

  • Azure Resource: Template Specs are stored as resources within a subscription.
  • Versioning: Supports versioning of your templates.
  • RBAC Control: Use Azure Role-Based Access Control (RBAC) to manage who can access and deploy the templates.
  • Simplified Sharing: Easier to share standardized templates across an organization compared to managing JSON files in repos or storage accounts.
