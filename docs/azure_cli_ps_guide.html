Command-Line Tools for Azure Management

Azure CLI and Azure PowerShell are cross-platform command-line tools for creating, managing, and automating Azure resources. Both provide powerful alternatives to the Azure Portal for scripting, automation, and DevOps workflows.

You (Admin)
CLI / PowerShell
Azure Resource Manager
Azure Resources

Azure CLI

  • Syntax: Uses az <group> <subgroup> <command> --parameter value structure
  • Cross-Platform: Runs on Windows, macOS, Linux, Docker, Azure Cloud Shell
  • Output: JSON by default; supports table, TSV, YAML formats
  • Querying: Uses JMESPath query language for filtering JSON output
  • Best For: Bash scripting, Linux environments, concise syntax

Azure PowerShell

  • Syntax: Uses Verb-AzNoun cmdlet structure (e.g., Get-AzVM, New-AzResourceGroup)
  • Cross-Platform: PowerShell Core 7+ runs on Windows, macOS, Linux
  • Output: Returns .NET objects (not text), enabling powerful pipeline manipulation
  • Querying: Uses Where-Object, Select-Object, and object properties
  • Best For: Windows environments, complex scripting, object-oriented workflows
Cloud Shell Advantage: Both tools are pre-installed in Azure Cloud Shell (accessible from portal or shell.azure.com), eliminating local installation and providing authenticated sessions automatically.

Connecting to Azure

Both tools authenticate via Azure Active Directory (AAD). After login, you can manage resources in your authorized subscriptions.

Azure CLI Authentication

Interactive Browser Login:

# Opens browser for AAD authentication
az login

# Login with specific tenant
az login --tenant <TENANT_ID>

# Check current account
az account show

Service Principal (for automation):

az login --service-principal \
  -u <APP_ID> \
  -p <PASSWORD_OR_CERT_PATH> \
  --tenant <TENANT_ID>

Switch Subscriptions:

# List subscriptions
az account list --output table

# Set active subscription
az account set --subscription "My Subscription Name"

PowerShell Authentication

Interactive Login:

# Opens browser for AAD authentication
Connect-AzAccount

# Login with specific tenant
Connect-AzAccount -Tenant <TENANT_ID>

# Check current context
Get-AzContext

Service Principal:

$credential = Get-Credential # Enter App ID as username, password/secret
Connect-AzAccount -ServicePrincipal \
  -Credential $credential \
  -Tenant <TENANT_ID>

Switch Subscriptions:

# List subscriptions
Get-AzSubscription

# Set active subscription
Set-AzContext -SubscriptionId <SUBSCRIPTION_ID>
AZ-104 Focus: Know the difference between az login vs Connect-AzAccount. For exam scenarios involving automation, recognize when service principal authentication is required (CI/CD pipelines, unattended scripts).

Common Operations: CLI vs PowerShell

Side-by-side examples for managing Resource Groups, Storage, VMs, and VNets.

Resource Groups

Operation Azure CLI Azure PowerShell
Create az group create --name MyRG --location eastus New-AzResourceGroup -Name MyRG -Location eastus
List az group list --output table Get-AzResourceGroup | Format-Table
Delete az group delete --name MyRG --yes Remove-AzResourceGroup -Name MyRG -Force

Storage Accounts

CLI:

# Create (name must be globally unique, 3-24 lowercase letters/numbers)
az storage account create \
  --name mystorageacct123 \
  --resource-group MyRG \
  --location eastus \
  --sku Standard_LRS

# List account keys
az storage account keys list \
  --account-name mystorageacct123 \
  --resource-group MyRG

PowerShell:

# Create
New-AzStorageAccount `
  -ResourceGroupName MyRG `
  -Name mystorageacct123 `
  -Location eastus `
  -SkuName Standard_LRS

# List keys
Get-AzStorageAccountKey `
  -ResourceGroupName MyRG `
  -Name mystorageacct123

Virtual Machines

CLI:

# Create Linux VM with SSH keys
az vm create \
  --resource-group MyRG \
  --name MyVM \
  --image Ubuntu2204 \
  --admin-username azureuser \
  --generate-ssh-keys \
  --size Standard_B2s

# Stop VM (deallocate to stop billing)
az vm deallocate --resource-group MyRG --name MyVM

# Get public IP
az vm show -g MyRG -n MyVM -d --query publicIps -o tsv

PowerShell:

# Create Windows VM
$cred = Get-Credential # Prompts for admin username/password
New-AzVm `
  -ResourceGroupName "MyRG" `
  -Name "MyVM" `
  -Location "East US" `
  -Image "Win2022Datacenter" `
  -Size "Standard_B2s" `
  -Credential $cred

# Stop VM
Stop-AzVM -ResourceGroupName MyRG -Name MyVM -Force

# Get public IP
(Get-AzPublicIpAddress -ResourceGroupName MyRG).IpAddress

Virtual Networks

CLI:

az network vnet create \
  -g MyRG \
  -n MyVNet \
  --address-prefix 10.0.0.0/16 \
  --subnet-name MySubnet \
  --subnet-prefix 10.0.1.0/24

PowerShell:

# Create subnet configuration first
$subnet = New-AzVirtualNetworkSubnetConfig `
  -Name MySubnet `
  -AddressPrefix "10.0.1.0/24"

# Create VNet with subnet
New-AzVirtualNetwork `
  -ResourceGroupName MyRG `
  -Name MyVNet `
  -Location eastus `
  -AddressPrefix "10.0.0.0/16" `
  -Subnet $subnet
Syntax Mnemonic - "CAP": CLI uses Commands with dashes (az group create). PowerShell uses Actions with verb-noun (New-AzResourceGroup). Both use Parameters (CLI: --name, PS: -Name).

Controlling Command Output

Both tools offer powerful filtering and formatting, but use different approaches.

Azure CLI Output

  • Output Formats:
    • --output json (default): Machine-readable JSON
    • --output table: Human-readable table view
    • --output tsv: Tab-separated values (script-friendly)
    • --output yaml: YAML format
  • JMESPath Querying: Use --query flag with JMESPath expressions

Examples:

# List VMs in East US region only
az vm list --query "[?location=='eastus'].{Name:name, RG:resourceGroup}" -o table

# Get first storage account name
az storage account list --query "[0].name" -o tsv

# Filter by tag
az resource list --query "[?tags.Environment=='Production'].name"

PowerShell Output

  • Object Pipeline: Cmdlets return .NET objects (not text), enabling property access and method calls
  • Filtering: Where-Object (alias: where, ?)
  • Selection: Select-Object (alias: select) to pick properties
  • Formatting: Format-Table (alias: ft), Format-List (alias: fl)

Examples:

# Filter VMs by location and select properties
Get-AzVM | Where-Object {$_.Location -eq 'eastus'} | Select-Object Name, ResourceGroupName

# Get storage accounts with Premium SKU
Get-AzStorageAccount | Where-Object {$_.Sku.Name -like 'Premium*'}

# Sort and format
Get-AzVM | Sort-Object Name | Format-Table Name, Location, @{Label="Size"; Expression={$_.HardwareProfile.VmSize}}
Real-World Scenario: Find all VMs that are running but not in the "Production" resource group:
CLI: az vm list -d --query "[?powerState=='VM running' && resourceGroup!='Production'].name"
PS: Get-AzVM -Status | Where {$_.PowerState -eq 'VM running' -and $_.ResourceGroupName -ne 'Production'} | Select Name
Exam Tip: CLI queries return JSON text filtered by JMESPath. PowerShell returns .NET objects filtered by Where-Object. Know which output format is best: table for humans, json/tsv for scripts.

Automating Azure Tasks

Both tools excel at automation. Use variables, loops, and error handling to create robust scripts.

Bash Scripting with CLI

#!/bin/bash
# Script: create-dev-environment.sh

set -e  # Exit on error

# Variables
RG_NAME="DevEnvironment"
LOCATION="eastus"
VM_NAME="DevVM"
VM_IMAGE="Ubuntu2204"

# Create resource group
echo "Creating resource group $RG_NAME..."
az group create --name $RG_NAME --location $LOCATION

# Create VM
echo "Creating VM $VM_NAME..."
az vm create \
  --resource-group $RG_NAME \
  --name $VM_NAME \
  --image $VM_IMAGE \
  --admin-username azureuser \
  --generate-ssh-keys

# Get public IP
PUBLIC_IP=$(az vm show -g $RG_NAME -n $VM_NAME -d --query publicIps -o tsv)
echo "VM created! Public IP: $PUBLIC_IP"

PowerShell Scripting

# Script: Create-DevEnvironment.ps1

param(
    [string]$ResourceGroupName = "DevEnvironment",
    [string]$Location = "East US"
)

$ErrorActionPreference = "Stop"  # Exit on error

try {
    # Create resource group
    Write-Host "Creating resource group $ResourceGroupName..." -ForegroundColor Cyan
    New-AzResourceGroup `
        -Name $ResourceGroupName `
        -Location $Location

    # Create storage account
    $StorageAccountName = "devstorage$(Get-Random -Maximum 9999)"
    Write-Host "Creating storage account $StorageAccountName..." -ForegroundColor Cyan
    $storage = New-AzStorageAccount `
        -ResourceGroupName $ResourceGroupName `
        -Name $StorageAccountName `
        -Location $Location `
        -SkuName Standard_LRS

    Write-Host "Environment created successfully!" -ForegroundColor Green
    Write-Host "Storage Account: $($storage.StorageAccountName)"
}
catch {
    Write-Error "Failed to create environment: $_"
    exit 1
}

Loops & Iteration

CLI (Bash loop):

# Create multiple storage accounts
for i in {1..3}; do
  az storage account create \
    --name "mystorage${i}$(date +%s)" \
    --resource-group MyRG \
    --location eastus \
    --sku Standard_LRS
done

PowerShell (ForEach loop):

# Create VMs in multiple regions
$regions = @("East US", "West US", "Central US")
foreach ($region in $regions) {
    $vmName = "VM-$($region.Replace(' ', ''))"
    New-AzVm `
        -ResourceGroupName "MultiRegionRG" `
        -Name $vmName `
        -Location $region `
        -Image "Win2022Datacenter"
}
Integration Tip: Use Azure CLI/PowerShell scripts in Azure DevOps Pipelines or GitHub Actions for CI/CD automation. Both tools integrate with ARM templates and Bicep for infrastructure deployment.

Decision Guide

Criteria Azure CLI Azure PowerShell
Syntax Style Command-based (az group create) Verb-Noun cmdlets (New-AzResourceGroup)
Output Type JSON text (parseable with JMESPath) .NET objects (use properties/methods)
Scripting Strength Bash/Linux scripting, concise one-liners Complex workflows, object manipulation, error handling
Learning Curve Lower (familiar to Linux admins) Moderate (PowerShell knowledge helpful)
Platform Preference Linux, macOS, cross-platform teams Windows environments, .NET developers
Cloud Shell Pre-installed (Bash mode) Pre-installed (PowerShell mode)
Community & Docs Extensive (Microsoft Learn, GitHub) Extensive (Microsoft Learn, PowerShell Gallery)

Choose Azure CLI When...

  • Working in Linux/Unix environments or Docker containers
  • Need concise, script-friendly commands for CI/CD pipelines
  • Prefer JSON output and JMESPath querying
  • Quick interactive commands in Cloud Shell (Bash mode)
  • Integrating with Bash scripts, Ansible, or Terraform

Choose Azure PowerShell When...

  • Working in Windows environments or have PowerShell expertise
  • Need complex object manipulation and pipeline filtering
  • Building enterprise automation with error handling and logging
  • Integrating with existing PowerShell scripts or modules
  • Leveraging .NET framework capabilities
Decision Mnemonic - "BOTH": Bash users CLI. Object-oriented needs PowerShell. Team preference matters. Hybrid approach is fineuse both!
Team Collaboration Tip: Many organizations use both tools. CLI for quick commands and Linux-based automation, PowerShell for Windows-centric workflows and complex scripts. Azure Cloud Shell supports bothswitch with pwsh (PowerShell) or bash commands.

Cross-Platform Command Line

A command-line tool for creating and managing Azure resources, installable on Windows, macOS, and Linux, or usable via Azure Cloud Shell.

  • Syntax: Uses a az <group> <subgroup> <command> --parameter <value> structure.
  • Cross-Platform: Consistent experience across different operating systems.
  • Shells: Works in Bash, PowerShell, cmd.exe.
  • Often preferred for its concise syntax and ease of scripting in Bash/Linux environments.

Cmdlets for Azure Management

A set of cmdlets (lightweight commands) for managing Azure resources directly from PowerShell, now cross-platform with PowerShell Core.

  • Syntax: Uses a Verb-AzNoun structure (e.g., `Get-AzResourceGroup`, `New-AzVM`).
  • Object-Oriented: Cmdlets often return .NET objects, allowing for powerful scripting and pipelining.
  • Modules: Functionality is organized into modules (e.g., `Az.Compute`, `Az.Storage`).
  • Traditionally strong in Windows environments, now available on macOS and Linux.

Controlling Command Output

CLI:

  • --output table: Human-readable table format.
  • --output json: Default, machine-readable JSON.
  • --output tsv: Tab-separated values (good for scripting).
  • --query "": Filter and shape JSON output using JMESPath queries. Example: `az vm list --query "[?location=='eastus'].name"`

PowerShell:

  • Format-Table (or `ft`): Displays output as a table. Use `-Property` to select columns.
  • Format-List (or `fl`): Displays output as a list of properties.
  • Select-Object (or `select`): Select specific properties.
  • Where-Object (or `where` or `?`): Filter objects.
  • Example: `Get-AzVM | Where-Object {$_.Location -eq 'eastus'} | Select-Object Name, ResourceGroupName`

Choosing the Right Tool

  • Azure CLI:
    • Often preferred by those comfortable with Bash/Linux environments.
    • Concise syntax can be quicker for simple commands.
    • Excellent JMESPath querying for JSON output.
  • Azure PowerShell:
    • Strong in Windows environments, integrates well with existing PowerShell scripts.
    • Object-oriented nature is powerful for complex scripting and data manipulation.
    • Mature module ecosystem.
  • Both are cross-platform now. Choice often comes down to personal preference, existing scripting skills, and team standards.
  • Both can be used in Azure Cloud Shell without local installation.
