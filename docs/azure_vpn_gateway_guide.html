Azure VPN Gateway Explained

Learn how VPN Gateways create secure, encrypted connections to your Virtual Network from on-premises or remote clients. A core hybrid connectivity topic for AZ-104.

Building Blocks & Operational Principles

An Azure VPN Gateway enables secure, encrypted traffic between your Azure Virtual Networks (VNets) and other locations over the public internet. It's a critical component for hybrid and multi-VNet connectivity.

Required Resources & Deployment Steps

  • Virtual Network (VNet): The VNet where your gateway will reside. Its address space must not overlap with the remote network.
  • GatewaySubnet: A special, dedicated subnet within your VNet, named exactly `GatewaySubnet`. Minimum size /27 recommended for future growth. You cannot use this subnet for other services.
  • Public IP Address: A Standard SKU Public IP (typically dynamic allocation for VNG) is used as the public endpoint for the gateway.
  • Virtual Network Gateway (VNG): The managed service deployed into the `GatewaySubnet`. This is where you specify the Gateway SKU and VPN Type. Azure manages the underlying VMs.
  • Local Network Gateway (LNG): An object that represents the "on-premises" or "remote" VPN device/network. It contains the remote public IP and address prefixes.
  • Connection: The logical resource that links the VNG to an LNG (or another VNG for VNet-to-VNet) using a shared key to establish the secure IPsec tunnel.

Key Constraint: You can only deploy one VPN Gateway per Virtual Network.

VPN Gateway Types (Policy-based vs. Route-based)

This is a critical choice made during gateway creation, dictating how traffic is handled and which connections are supported. It cannot be changed later.

Feature Policy-based VPN (Legacy) Route-based VPN (Recommended)
Mechanism Encrypts traffic based on a *static list* of source/destination address prefixes (a "policy" or traffic selector). Encrypts traffic based on dynamically *routed* IP prefixes. IPsec tunnels are modeled as virtual interfaces.
Authentication IKEv1 only. IKEv2.
Routing Uses *static routing*. Addresses behind the tunnel are explicitly defined. Uses *dynamic routing protocols* (like BGP). IPsec tunnels can carry traffic from any-to-any address.
Connections Supported Site-to-Site (S2S) only. Limited features and connections. Site-to-Site (S2S), Point-to-Site (P2S), VNet-to-VNet, and ExpressRoute coexistence. Highly flexible.
Common Use Compatibility with older on-premises VPN devices. Standard for almost all modern, flexible, and scalable hybrid connections.

Gateway SKUs & Performance

The SKU (size) determines the gateway's capabilities, throughput, and pricing. Higher SKUs provide more bandwidth and connections.

  • The `Basic` SKU is **not recommended for production** workloads and has limitations (e.g., no Zone-Redundancy, no ExpressRoute coexistence).
  • SKUs like `VpnGw1`, `VpnGw2`, up to `VpnGw5` offer increasing throughput (from ~650 Mbps to ~10 Gbps).
  • Gateway generation (Gen1 vs. Gen2) also impacts features and performance. Gen2 offers significantly improved performance and features.

High Availability & Redundancy Options

  • Active/Standby (Default): Two instances deployed, one active, one standby. If the active instance fails or undergoes planned maintenance, standby takes over. Causes brief interruption (~30-90 seconds).
  • Active/Active: Both gateway instances are active and establish tunnels. Traffic is load-balanced across both tunnels. Requires BGP. Provides better throughput and faster failover (minimal interruption).
  • Zone-Redundant Gateways: Deploys gateway instances across different Availability Zones within a region. Protects against datacenter-level failures. Requires `...AZ` Gateway SKUs (e.g., `VpnGw1AZ`).
  • ExpressRoute Failover: A Site-to-Site VPN Gateway can be configured as a secure failover path for an ExpressRoute connection. If the private ExpressRoute circuit fails, traffic can be rerouted over the public internet via the VPN tunnel.

Which VPN Connection to Use?

Choosing the right VPN connection type depends on what you are connecting to your Azure VNet. This is a common point of confusion.

Connection Type Connects What? Primary Use Case
Site-to-Site (S2S) An entire on-premises network (e.g., your corporate office) to an Azure VNet. Extending your corporate network to Azure. All servers on-prem can potentially reach Azure resources.
Point-to-Site (P2S) An individual client computer (e.g., a developer's laptop) to an Azure VNet. Allowing remote workers or administrators to securely connect to Azure from anywhere, like a coffee shop or home office.
VNet-to-VNet One Azure VNet to another Azure VNet. Connecting two VNets using the VPN Gateway infrastructure. Note: VNet Peering is often the preferred method for this.

Key Differentiators & "Gotchas":

  • Site-to-Site (S2S): Think "Site" = Office or Datacenter. It's a permanent link between networks. Requires a VPN device on-premises.
  • Point-to-Site (P2S): Think "Point" = A single computer/laptop. Requires client VPN software to be installed on that computer.
  • VNet-to-VNet vs. VNet Peering:
    • Use VNet Peering first for connecting VNets. It's simpler, has higher performance, and stays on the Azure backbone.
    • Use VNet-to-VNet (via VPN Gateway) only if you need transitive routing (e.g., on-prem -> VNetA -> VNetB), which peering doesn't support without extra configuration.

AZ-104 Focus: Be able to read a scenario and determine if it requires a Site-to-Site or Point-to-Site connection. Understand that VNet Peering is usually the better choice for connecting VNets in Azure.

Practical Steps & Key Learnings

Your lab simulated creating a Site-to-Site VPN connection between two Azure VNets. Let's review the components and key learnings.

Resources Created (The Building Blocks)

  • Virtual Networks & Subnets: Two VNets (`Azure-VNet-1`, `HQ-Network`).
    • Crucially, each included a subnet named `GatewaySubnet`. This is a mandatory, exact name.
  • Public IPs: A Standard SKU, Static Public IP was created for each Virtual Network Gateway.
  • Local Network Gateways (LNGs): Represents the "other side" of the connection. An LNG holds the public IP and address space of the remote network.
  • Virtual Network Gateways (VNGs): The actual gateway service deployed into each VNet's `GatewaySubnet`. This resource performs the VPN work.
  • Connections: The logical object that links a VNG to an LNG (or another VNG for VNet-to-VNet) using a pre-shared key to establish the secure tunnel. (This was the final step noted in your lab).

Key Learnings & AZ-104 Exam Pointers

  • `GatewaySubnet` is Mandatory: It must be named exactly `GatewaySubnet`.
  • SKU Requirement: Gateways require a specific Gateway SKU (`VpnGw1`, `VpnGw2AZ`, etc.). The `Basic` SKU is not for production and has limitations. The Public IP used must be `Standard` SKU for zone-aware gateways.
  • LNG Represents "The Other End": A Local Network Gateway is a representation of the remote site's configuration.
  • Connection is Two-Way: A connection must be established from each gateway to the other.
  • VPN Type: `RouteBased` is the modern, flexible VPN type and is required for most scenarios. `PolicyBased` is a legacy type for specific on-prem devices.

Exam Essentials

  • A VPN Gateway requires a dedicated subnet named **`GatewaySubnet`**. This is a mandatory naming convention.
  • A **Local Network Gateway (LNG)** is a resource that represents your on-premises network's configuration (Public IP and address space).
  • A **Connection** resource uses a shared key to link the VNG and LNG, establishing the secure tunnel.
  • Differentiate **Site-to-Site (S2S)** (for connecting networks) from **Point-to-Site (P2S)** (for connecting individual computers).
  • Remember that for VNet-to-VNet connectivity, **VNet Peering is preferred** over using a VPN Gateway due to better performance and simplicity.
  • Understand the high-availability options: **Active/Standby** (default) and **Active/Active** (requires BGP).
  • Gateway **SKUs** determine performance and features. **AZ SKUs** are used for zone-redundant deployments.
  • Know the difference between **Policy-based VPNs** (legacy, static routing, limited support) and **Route-based VPNs** (modern, dynamic routing, recommended, supports all connection types).
