Azure Monitor: Alerting & Management

Master Azure Monitor's capabilities for collecting telemetry, detecting issues with various alert types, and automating responses. Essential for AZ-104.

Comprehensive Telemetry for Cloud & On-Premises

Azure Monitor is a native Azure service designed for collecting, analyzing, and acting on telemetry data from your cloud and on-premises environments. It helps you understand the performance and availability of your applications and infrastructure.

Key Features & Availability:

  • Immediate Availability: Azure Monitor is immediately available when you create an Azure subscription.
  • Built-in Data:
    • Activity Log: Automatically collects events about activity in your Azure subscription (e.g., VM creation, deletions).
    • Platform Metrics: Automatically collected for any Azure resources you create, providing numerical time-sensitive values (e.g., CPU, memory usage).
    • Metrics explorer is available to analyze this data right out of the box.
  • Requires Configuration:
    • Diagnostic Settings: Needed to collect detailed logs and metrics from resources and send them to destinations like Log Analytics or Storage Accounts.
    • Alerts: Must be configured to notify you when important conditions are met (e.g., high CPU, errors).
  • Access & Configuration Methods:
    • Azure Portal (via the central "Monitor" menu or "Monitoring" section within specific service menus).
    • Azure CLI, PowerShell, and REST API for automation and scripting.

AZ-104 Focus: Understand Azure Monitor as the central monitoring service. Differentiate between features available out-of-the-box (Activity Log, Platform Metrics) and those requiring configuration (Diagnostic Settings, Alerts).

Signals Driving Your Alerts

Azure Monitor collects various types of telemetry. The nature of the resource dictates which data types are available for monitoring and alerting.

Data Types in Azure Monitor:

  • Metrics:
    • Numerical, time-sensitive values representing an aspect of the target resource.
    • Focus: Real-time performance (e.g., CPU percentage, network in/out).
  • Logs:
    • Content data held in structured, record-based log files.
    • Focus: Historical trends and patterns (e.g., web server access logs, application errors). Requires Log Analytics Workspace.
  • Activity Log:
    • Records events that occur in the Azure subscription, like resource creation, deletion, or configuration changes.
    • Focus: Changes to resources or service health events.

Primary Alert Types (based on Signal Type):

Azure Monitor supports three main signal types that can be used to monitor your environment and trigger alerts:

  • Metric Alerts: Trigger based on real-time numeric thresholds (e.g., CPU > 90%, disk I/O exceeding a limit).
  • Log Search Alerts: Trigger based on queries over log data stored in Log Analytics (e.g., more than 500 HTTP 404 errors in the past hour, repeated failed login attempts).
  • Activity Log Alerts: Trigger on Azure resource events recorded in the Activity Log (e.g., a Virtual Machine is deleted, a new role assignment occurs).
Metric = Real-time threshold (CPU, memory)
Log = Trends/patterns from logs (500 errors, login attempts)
Activity Log = Change detection (resource events)

AZ-104 Focus: Be able to differentiate the three main alert types based on their underlying data source and their best use cases (real-time, historical analysis, or resource changes).

Monitoring Real-time Numerical Data

Metric alerts are used for regular threshold monitoring of Azure resources. They send notifications when predefined numeric conditions are met on metric values.

Condition Types:

  • Static Thresholds:
    • You define a fixed threshold (e.g., "if CPU > 85%").
    • The alert triggers when the metric value crosses this static threshold for a specified duration.
    • Example: "if CPU > 85% for 10 minutes, trigger."
  • Dynamic Thresholds:
    • Uses machine learning to adapt thresholds based on historical metric behavior.
    • No fixed threshold; the system learns normal patterns and alerts on significant deviations.
    • Parameters like "look-back period" (how many previous periods to evaluate) and "number of violations" (how many deviations before firing) define the sensitivity.

Scope & Dimensions:

  • Dimensions: Allow you to monitor data from multiple target instances with a single metric alert rule.
    • Example: Monitor CPU utilization for all VMs in a group by using the asterisk (*) wildcard, or by naming specific instances.
  • Scaling Metric Alerts: Apply one metric alert rule across many resources (e.g., all virtual machines in a subscription or resource group). A single alert rule can monitor resources within one Azure region.

Use Cases: Ideal for real-time performance monitoring like high CPU usage, low disk space, high network latency, or exceeding request rates on a web app.

AZ-104 Focus: Differentiate static vs. dynamic thresholds. Understand how dimensions enable monitoring multiple instances with a single rule. Recognize metric alerts as the primary choice for real-time numeric performance issues.

Detecting Patterns & Trends in Log Data

Log search alerts trigger based on Kusto Query Language (KQL) queries over log data stored in a Log Analytics Workspace. They are ideal for detecting historical trends or specific patterns in your application and infrastructure logs.

Use Cases:

  • Detecting patterns in historical data, such as repeated 500 errors from a web server.
  • Spotting security events like multiple failed login attempts from a single IP.
  • Analyzing complex application errors that appear across multiple log entries.

Rule Components:

  • Log Query: The KQL query that runs periodically.
  • Time Period: The time range over which the query evaluates data (e.g., "in the last 30 minutes").
  • Frequency: How often the log query should run (e.g., "every 5 minutes").
  • Threshold: The trigger point for an alert to be created.

Types of Log Search Alerts:

  • Number of Records: Triggers if the query returns a certain number of records within the time period (e.g., "if query returns > 5 failed logins").
  • Metric Measurement: Aggregates values from the query results and triggers based on a numeric threshold (e.g., "if average CPU usage (from logs) > 80%"). This type can also use:
    • Aggregate Function: Calculation (count, average).
    • Group Field: How results are grouped (e.g., by computer name).
    • Interval: Time aggregation window.

Stateful vs. Stateless:

  • Stateless: Alerts fire each time the condition is met, even if fired previously (can lead to alert spam if not handled carefully).
  • Stateful: Alerts fire when the conditions are met but will not fire again or trigger more actions until the conditions are resolved.

AZ-104 Focus: Understand log search alerts are for historical data analysis and patterns. Know the key rule components (query, time period, frequency, threshold). Differentiate between "Number of Records" and "Metric Measurement" types.

Detecting Resource Changes & Service Health Events

Activity log alerts notify users of specific events within an Azure subscription, such as resource changes or service health issues. They are based on events recorded in the Azure Activity Log.

Purpose & Alert Types:

  • Purpose: Detect changes to Azure resources or important service health notifications.
  • Specific Operations: Triggers when an Azure resource undergoes a specific operation.
    • Example: A VM is created, deleted, powered off, or a new role is assigned to a user.
    • Scope can be subscription, resource group, or individual resource level.
  • Service Health Events: Notifies about incidents, planned maintenance, health advisories, or security advisories impacting Azure services in a particular region or globally.

Trigger Conditions:

You can configure alerts based on various properties of Activity Log events:

  • Category: (e.g., Administrative, Service Health, Autoscale, Policy, Recommendation).
  • Resource Type: (e.g., `Microsoft.Compute/virtualMachines`).
  • Operation Name: (e.g., `Microsoft.Compute/virtualMachines/delete`).
  • Level: (e.g., Verbose, Informational, Warning, Error, Critical).
  • Status: (e.g., Started, Failed, Succeeded).
  • Event Initiated By: (e.g., the email address or Microsoft Entra ID of the user who initiated the event).

Key Point: Activity log alerts monitor events only in the subscription where they are created. Service health alerts are typically for a whole region, so no specific resource selection is needed for them.

AZ-104 Focus: Understand Activity Log alerts are for monitoring resource changes and service health. Know key trigger conditions like operation name, resource type, and event initiator. Differentiate them from metric and log alerts.

The Anatomy of an Alert Rule

Every alert or notification in Azure Monitor is the product of an alert rule. The composition of an alert rule remains consistent across all alert types.

Core Alert Rule Components:

  • Resource (Scope): The target resource(s) being monitored by the alert rule (e.g., a specific VM, a resource group, a subscription). You can assign multiple target resources to a single alert rule. The resource type defines the available signal types.
  • Condition:
    • The signal type used to assess the rule (Metric, Log, or Activity Log).
    • The alert logic applied to the data supplied by the signal type (e.g., `CPU > 80%`, `query results > 5`).
  • Actions: What happens when the alert fires. Typically points to an Action Group.
  • Alert Details:
    • Alert Name + Description: Define the alert's purpose.
    • Severity Level: Indicates the criticality of the alert. (0: Critical, 1: Error, 2: Warning, 3: Informational, 4: Verbose).

Managing Alert Rules:

  • Alert rules can be enabled or disabled as needed (e.g., disable alerts during a planned deployment).
  • The Alerts summary view in the Azure portal provides a central place to see all fired alerts, filter them by subscription, severity, condition, or time range.

Alert Condition States:

  • When an alert triggers, its monitor condition is set to "Fired."
  • After the underlying condition that caused the alert to fire clears, the monitor condition is set to "Resolved."

AZ-104 Focus: Know the five core components of any alert rule. Understand the five severity levels. Be familiar with managing alert rules (enable/disable) and the "Fired" vs. "Resolved" states.

What Happens When an Alert Fires

An Action Group is a reusable collection of notification preferences and actions that can be triggered by any Azure Monitor alert (Metric, Log Search, Activity Log, Azure Service Health, Azure Advisor alerts).

Definition & Reusability:

  • A reusable group of recipients and actions.
  • Once created, an action group can be reused across multiple alert rules, simplifying management.
  • You can add an action group to an alert rule during its creation or by editing an existing alert.

Examples of Actions:

  • Notifications: Send an email, SMS message, push notification to Azure mobile app, or make a voice call.
  • Automation: Trigger a Logic App, Azure Function, Webhook, Azure Automation Runbook (e.g., to restart a VM or scale a VM).
  • Integration: Create an IT Service Management (ITSM) ticket.

AZ-104 Focus: Understand Action Groups as reusable sets of notifications and automation actions. Know common examples of actions (email, webhook, runbook, Logic App).

Modifying Alert Behavior at Scale

Alert Processing Rules allow you to modify the normal behavior of alerts *after* they are fired. They are used to add or suppress action groups, enabling alert management at scale.

Purpose & Use Cases:

  • Suppress Notifications: The most common use case is to suppress notifications during planned maintenance windows to prevent alert storms.
  • Add Actions: Add an action group to alerts that match certain criteria (e.g., send all critical alerts for a specific application to a dedicated paging system).
  • Apply Filters: Apply rules to specific resource scopes (single resource, resource group, entire subscription) or filter by alert properties (e.g., apply only to alerts with Severity 0).
  • Scheduling: Rules can be configured with a one-time or recurring schedule (e.g., suppress notifications every Sunday 2 AM - 4 AM for database maintenance). By default, rules are always active.
Processing Rule = Override behavior (suppress, reroute)

Key Distinction: Alert rules *trigger* alerts when conditions are met. Alert processing rules *modify* those alerts as they are being fired.

AZ-104 Focus: Understand Alert Processing Rules' primary function (suppression during maintenance, adding/removing actions at scale). Know they operate *after* an alert is fired and support scheduling/filters.

Deeper Monitoring for Virtual Machines

VM Insights is a feature of Azure Monitor that provides a comprehensive view of the performance and health of your Azure virtual machines (VMs) and Virtual Machine Scale Sets (VMSS). It works across Azure, hybrid environments, and even other clouds.

How VM Insights Works:

  • Agent Deployment: Deploys the Azure Monitor Agent (AMA) to your VMs. Optionally, the Dependency Agent is deployed to enable the "Map" view (showing process and network dependencies).
  • Data Collection Rules (DCRs): Uses Data Collection Rules, which are tied to a Log Analytics workspace, to define what performance data and logs are collected from the VMs.

What You Get (Key Insights):

  • Prebuilt Performance Charts: Visualizations for key performance counters like CPU utilization, available memory, logical disk performance, and network I/O. Helps quickly spot anomalies and bottlenecks.
  • Service Map & Dependency Visuals: Provides a visual map of interconnected processes and active network connections on and between your VMs. Crucial for understanding application dependencies and troubleshooting connectivity.
  • Guest OS Health: Monitors the health of the guest operating system and the processes running within it.

Enablement:

  • From the Azure portal, navigate to "Monitor" > "Virtual Machines".
  • You can then enable monitoring directly for "Not Monitored" VMs.

AZ-104 Relevance: Knowing VM Insights is essential for deeper VM monitoring capabilities, configuring performance alerts, and understanding application dependencies through the Service Map. Exam expects awareness of enabling it, agent deployment, and the types of insights it provides.

AZ-104 Focus: Understand VM Insights as a specialized Azure Monitor feature for VMs. Know it uses AMA/Dependency Agent and DCRs tied to Log Analytics. Be familiar with its benefits (performance charts, Service Map).

Diagnosing Network Health & Connectivity

Azure provides tools within Azure Monitor (Network Insights) and dedicated Network Watcher services to help you monitor, diagnose, and troubleshoot network performance and connectivity issues across your Azure infrastructure.

Network Insights (in Monitor Blade):

  • Shows high-level health and metrics for various networking resources.
  • Filters by resource type (NICs, NSGs, Route Tables, Public IPs, Load Balancers, Virtual Networks) and health state (Available, Degraded, Unavailable, Unknown).
  • Displays connectivity metrics, especially for hybrid VMs if the Azure Monitor agent is installed.

Network Watcher Tools (Essential for AZ-104):

Network Watcher offers a suite of diagnostic and visualization tools for your Azure network.

  • 1. IP Flow Verify:
    • Tests if a TCP or UDP packet is allowed or denied by NSG rules.
    • Crucially, it shows the blocking rule and its name if traffic is denied.
    • Used for quick checks of network access configuration.
  • 2. Connection Troubleshoot:
    • Checks end-to-end connectivity (TCP/ICMP) between a source VM and a destination (another VM, IP address, FQDN, or URI).
    • Diagnoses the root cause of connection failures (e.g., NSG, User Defined Routes (UDRs), firewall, DNS resolution issues).
  • 3. Next Hop:
    • Reveals the next hop IP address and type for a packet destined for a specific IP address from a given VM.
    • Helps determine the routing path (VNet local, UDR, Internet, Virtual Appliance).
  • 4. Effective Security Rules:
    • Aggregates all NSG rules (from both the NIC and subnet NSGs) applied to a specific network interface.
    • Shows the actual security posture for a VM, helping to understand effective allow/deny rules.
  • 5. Packet Capture:
    • Captures raw network packets from a VM's Network Interface (NIC).
    • The captured data is downloadable as a .pcap file for deep analysis with tools like Wireshark.
  • 6. VPN Troubleshoot:
    • Diagnoses common VPN Gateway errors and checks tunnel health.
    • Provides metrics like packet drops and bandwidth usage for VPN connections.
  • 7. Topology:
    • Generates a visual network diagram.
    • Shows the interconnections between resources like VNets, subnets, NSGs, NICs, VMs, and VNet peering, aiding visual understanding of your network layout.

AZ-104 Relevance: Network Watcher tools are crucial. IP Flow Verify and Connection Troubleshoot are frequently tested for diagnosing connectivity issues. Understanding all listed tools and their primary use cases is important for the exam.

AZ-104 Focus: Recognize Network Watcher as the go-to service for network diagnostics. Know the purpose of each key tool, especially IP Flow Verify, Connection Troubleshoot, Next Hop, and Effective Security Rules.

Hands-on: Stressing a VM & Alerting on CPU

This example demonstrates how to create a Linux VM that stresses its CPU, and then set up a metric alert using Azure CLI to notify you when CPU usage exceeds a threshold.

1. Create cloud-init.txt for VM Stressing:

This script installs `stress` and runs it to generate high CPU utilization.

cat < cloud-init.txt #cloud-config package_upgrade: true packages: - stress runcmd: - sudo stress --cpu 1 EOF

2. Create the Linux VM:

Deploys an Ubuntu VM and attaches the `cloud-init.txt` script.

az vm create \ --resource-group "YOUR_RESOURCE_GROUP" \ --name vm1 \ --location eastus \ --image Ubuntu2204 \ --custom-data cloud-init.txt \ --generate-ssh-keys

3. Obtain VM Resource ID:

The alert command requires the full Azure Resource ID of the VM.

VMID=$(az vm show --resource-group "YOUR_RESOURCE_GROUP" --name vm1 --query id -o tsv)

4. Create a Metric Alert for CPU Usage:

This command creates a metric alert named "Cpu80PercentAlert" that triggers when the VM's CPU percentage (max aggregation) exceeds 80% for at least 1 minute, evaluated every 1 minute, with a severity of 3 (Informational).

az monitor metrics alert create \ -n "Cpu80PercentAlert" \ --resource-group "YOUR_RESOURCE_GROUP" \ --scopes $VMID \ --condition "max percentage CPU > 80" \ --description "Virtual machine is running at or greater than 80% CPU utilization" \ --evaluation-frequency 1m \ --window-size 1m \ --severity 3

5. View Alerts in Azure Monitor:

After a few minutes, as the VM's CPU gets stressed, you should see the alert fire in the Azure portal:

  • Navigate to Monitor > Alerts in the Azure portal.
  • You will see "Cpu80PercentAlert" (and potentially a "Cpu90PercentAlert" if you configured that in the portal as per the full lab).
  • The alert condition will change from "New" to "Fired" once the CPU threshold is breached.

Note: It can take up to 10 minutes for a new metric alert rule to become active and for the alert to show up after the condition is met.

AZ-104 Focus: Practice translating alert requirements into CLI commands. Understand the parameters for `az monitor metrics alert create` (name, scopes, condition, description, evaluation frequency, window size, severity).

Exam Essentials & Alerting Strategy

  • Azure Monitor Basics: It's Azure's native monitoring service. Activity Log & Platform Metrics are built-in; Diagnostic Settings & Alerts require configuration.
  • Three Main Alert Types & Use Cases:
    • Metric Alerts: Real-time numerical data (CPU, memory). Use static or dynamic thresholds.
    • Log Search Alerts: Historical trends/patterns in log data (500 errors, failed logins). Requires KQL queries.
    • Activity Log Alerts: Changes to Azure resources or service health events (VM deleted, new role assignment).
  • Alert Rule Components: Resource (scope), Condition (signal type + logic), Actions (Action Group), Severity (0-4), Name/Description.
  • Action Groups: Reusable collection of notifications (email, SMS, voice) and actions (Logic App, Webhook, Runbook) to be triggered by an alert. Remember: "Action Group = What happens next."
  • Alert Processing Rules: Modify behavior of *fired* alerts (e.g., suppress notifications during maintenance, add/remove actions globally). They *override* normal behavior. Remember: "Processing Rule = Override behavior (suppress, reroute)."
  • VM Insights: A key Azure Monitor feature for VMs and VMSS. Uses Azure Monitor Agent (AMA) and Data Collection Rules (DCRs) with Log Analytics. Provides performance charts and Service Map (for dependencies). Essential for deeper VM monitoring.
  • Network Monitoring & Troubleshooting: Utilizes Network Insights for high-level health and Network Watcher tools (IP Flow Verify, Connection Troubleshoot, Next Hop, Effective Security Rules, Packet Capture, VPN Troubleshoot, Topology) for in-depth diagnostics. Crucial for network connectivity and security validation.
  • CLI Usage: Be familiar with creating VMs with custom data (`cloud-init.txt`) and configuring metric alerts (`az monitor metrics alert create`) via CLI.

**Exam Tip:** Pay close attention to the *scenario* when deciding which alert type to use. Is it a live performance issue (Metric)? A historical pattern (Log Search)? A resource configuration change (Activity Log)? Or a network connectivity issue (Network Watcher)?
